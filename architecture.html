v0.2.0 · 工具无关层（你的反馈点 #1）

🎯 你提的问题
当前设计把项目工作流（agent 定义、流程、规则、技能）放进了 .claude/ —— 这是 Claude Code 专用目录。 如果以后换工具（Cursor / Copilot / 自研），这些资产就废了。项目文档应该跟着 git 走，而不是绑定到某个 IDE。

设计方案

当前 v0.1.0 — 与 Claude Code 耦合
your-project/
├── .claude/              ← Claude Code 专用
│   ├── agents/*.md       ← 角色契约
│   ├── skills/*/SKILL.md ← 标准操作
│   └── settings.json
├── CLAUDE.md             ← Claude Code 读
└── docs/workflow.md
    docs/dev-map.md       ← 这些已经工具无关
    docs/tasks.md
    docs/spec/
v0.2.0 — 工具无关 + 绑定生成
your-project/
├── .harness/             ← 真相源（工具无关）
│   ├── agents/*.md       ← 角色契约（纯 Markdown）
│   ├── skills/*/SKILL.md
│   ├── workflow.md
│   ├── rules/            ← 规则源
│   │   ├── core.md
│   │   └── stack.md
│   └── bindings/         ← 工具绑定
│       └── claude-code.yaml
├── .claude/              ← 生成（可 .gitignore）
│   ├── agents/   ← sync from .harness/
│   ├── skills/   ← sync from .harness/
│   └── settings.json
├── CLAUDE.md             ← 从 .harness/rules 合成
└── docs/...              ← 不变

核心变化

层角色谁来维护Git 行为
.harness/真相源，工具无关 Markdown / YAML开发者直接编辑追踪
.claude/Claude Code 绑定层（生成）不要手改；用 harness sync可选追踪（建议追踪以便团队成员开箱即用，但允许 .gitignore）
CLAUDE.md从 .harness/rules/ 合成的 Claude Code 入口不要手改追踪
docs/项目文档（工具无关）开发者追踪

新增的同步流程

# 任何时候改了 .harness/，跑一下同步
$ harness sync
🔄 .harness/agents/ → .claude/agents/
🔄 .harness/skills/ → .claude/skills/
🔄 .harness/rules/ + workflow.md → CLAUDE.md
✓ Synced. Claude Code will pick up changes on next session.

# 未来支持 Cursor
$ harness sync --binding=cursor
🔄 .harness/rules/ → .cursor/rules/*.mdc
🔄 .harness/agents/ → Cursor Composer config
...

权衡

✅ 收益

  • 换 IDE 不丢资产 — 核心知识不跟 Claude Code 走
  • 多工具共存可行（不同团队成员用不同 IDE）
  • 角色契约、流程定义这些"项目知识"变成一等公民，与代码同等地位
  • 未来 verify_all 可以加 .harness/.claude/ 一致性检查（类似当前 sync-self）

❌ 代价

  • 多一层间接：用户必须知道 "编辑 .harness/，不要直接动 .claude/"
  • 需要 harness sync 命令（可做成 Skill 或 hook）
  • v0.1 装好的项目需要迁移（写迁移脚本）
  • 编辑 .harness/agents/*.md 后忘 sync 容易踩坑（hook 自动 sync 可缓解）

💡 实现要点

这其实就是把本仓库现有的"templates/common/.claude/agents/ 是 SOT，根 .claude/ 是副本"思路下沉到用户项目层。 本仓库已经在用类似 sync-self 机制，所以工程模式是验证过的，只需复刻到 init 模板。

v0.3.0 · Developer 分区（你的反馈点 #2）

🎯 你提的问题
当前的 Developer 是单一 generic agent，不知道项目的实际分区。集成到具体项目时， 应该由 AI 根据项目结构（前端/后端/DB/微服务……）创建多个分区 Developer，每个负责自己的部分。 这样 agent 才更贴合项目。

每个分区 Developer 只读自己负责的文件，更专注。Architect 在方案阶段就标好哪部分由哪个分区做。

分区方案（按项目类型）

项目类型典型分区 Developer 提议分区策略
Fullstack (Next + Nest + Postgres) dev-frontend / dev-backend / dev-db 按 monorepo 包或顶层目录（apps/web, apps/api, prisma）
Backend (微服务) 每个服务一个：dev-orders / dev-users / dev-billing 按 service 目录
Backend (单体 API) dev-api / dev-services / dev-db 按层（controller/service/repository）
Fullstack (无明显分区) 退化为单一 developer（保留 v0.1 行为） 不强行分区

新工作流

# init 时
> /harness-init
  Type? → fullstack
  Stack? → Next.js + NestJS + Postgres
  Detected monorepo layout: apps/web, apps/api, prisma/
  Suggested developer partitions:
    - dev-frontend  (owns: apps/web/**)
    - dev-backend   (owns: apps/api/**)
    - dev-db        (owns: prisma/**)
  Accept? [y/edit/n]   ← 用户可以编辑

# 任务派发时
> Take this task: Add CSV export to orders page (UI + API + DB)

  PM Orchestrator:
    Architect produces 02_SOLUTION_DESIGN.md with partition assignments:
      - dev-frontend: add button + download handler
      - dev-backend:  add GET /api/orders/export.csv
      - dev-db:       no migration needed
    Dispatching in dependency order: dev-db → dev-backend → dev-frontend

权衡

✅ 收益

  • 每个 Developer 的上下文聚焦，context 利用率提升
  • 专业性更强（前端 dev 不需要懂 SQL migration）
  • 潜力：跨分区并行（多脑多手）
  • 分区即文档：dev-map 自然衍生

❌ 代价

  • 跨分区任务（如 CSV 导出）需要 Architect 显式拆解 — 多一步
  • 分区边界要清晰，否则推诿（"那个文件应该归谁"）
  • 小项目可能过度设计 — 所以要支持退化为单一 Developer
  • 整套 agent 数量从 7 涨到 9-12，复杂度上升

实现要点（如果做）

  1. init 阶段加分区检测：扫顶层目录、package.json workspaces、monorepo 配置 → 提议分区
  2. 分区契约：每个 dev-* 的 SKILL.md 写明 owns: 路径 glob
  3. Architect 升级：方案阶段必须为每个 in-scope 文件指定责任 Developer
  4. PM 路由：按 Architect 的 partition 分配执行；支持并行（默认串行）
  5. 退化机制：项目小 / 无明显分区 → 创建单一 developer，等同 v0.1 行为
  6. verify_all 加项：检查每个分区 Developer 的 owns glob 不重叠

我的建议（你来定）

选项说明工作量
A · 先做 v0.2 工具无关层，再 v0.3 Developer 分区 先解决"换工具会废"这个更结构性的问题，分区在工具无关基础上做更稳。 大（v0.2 大概 4-6 commits；v0.3 类似）
B · 先 v0.3 Developer 分区，再 v0.2 工具无关 分区收益更直接可见；但之后做工具无关时要把分区结构二次迁移。 大 + 返工
C · 两个一起做（v0.2+v0.3 合并发布） 一次到位，但 commit 历史会复杂，单次改动面大。 很大
D · 先稳 v0.1，把没端到端测过的实际跑通，再启 v0.2 把"安装 + 真实 Claude Code 跑一次"作为 v0.1.1 验证步，再启 v0.2。 小（测试为主）

💡 我推荐 A

理由：
  • v0.2 解决的是"方向性"问题（结构性绑定），v0.3 解决的是"贴合度"问题（局部优化）。先把方向定准，再优化局部更稳。
  • v0.3 在工具无关基础上做，相当于在 .harness/agents/ 加分区文件 + 修改 init 提议逻辑，比绑死在 .claude/ 上做要清爽。
  • v0.1 已经验证了 7 agent + verify_all 的工程模式，v0.2 主要是"把这套搬到 .harness/"，工程模式不变，风险可控。
但你说不算就不算，等你决策。

关键纪律

  • 发现问题的人不能修问题（Reviewer 不改代码、Gate 不改需求）
  • 下游不能改上游文档，只能提阻塞回退给 PM
  • 同阶段 ≥3 次回退 → 停下来问人
  • PM 只做路由，绝不给专业建议
  • Planner 输出声明式计划（JSON），不直接调用
  • baseline 只升不降 — 测试数减少 = FAIL

Rule / Skill / Script 三层

层位置性质用法
RuleCLAUDE.md软约束（自然语言）原则 / 红线 / always-apply ≤5 条
Skill.claude/skills/*/SKILL.md标准操作手册编译/测试/验证等固定动作
Scriptscripts/verify_all.{ps1,sh}硬门禁可机器判定的全下沉

原则：能用机器查的就别靠 AI 记。Rule 越多越容易被忽略，越下沉越靠谱。
