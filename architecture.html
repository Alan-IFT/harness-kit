🔄 Two-layer consistency

实现要点

  1. init 阶段加分区检测：扫顶层目录、package.json workspaces、monorepo 配置 → 提议分区
  2. 分区契约：每个 dev-* 的定义里写明 owns: 路径 glob
  3. Architect 升级：方案阶段必须为每个 in-scope 文件指定责任 Developer
  4. PM 路由：按 Architect 的 partition 分配执行；支持并行（默认串行）
  5. 退化机制：项目小 / 无明显分区 → 单一 developer
  6. verify_all 加项：检查每个分区 Developer 的 owns glob 不重叠
  7. v0.3 同时实现 harness-adopt 自动 apply（adopt 一直是 v0.3 承诺的功能）

为什么不一开始就完全依赖 AI？

🎯 你提的问题（很关键）
现在的设计依赖代码（verify_all 脚本、harness-sync 脚本、模板复制 + 占位符替换、test-init 等）。 未来 AI 更新时，这些代码也要更新维护。为什么不一开始就设计成"AI 完成一切"？

简短答案

我们做的是"代码 + AI"混合架构。代码做机械的、可重复的工作；AI 做需要判断的工作。 这不是因为"不敢全 AI"，而是因为某些事情代码做得更可靠、更便宜、更可测。 但你的反思有价值——确实有一些代码可以转 AI，会在 v0.5+ 渐进 AI 化。

当前"代码 vs AI"边界

责任谁做为什么
读 SPEC，写需求文档AI（requirement-analyst agent）需要语义理解，代码做不了
设计技术方案AI（solution-architect）需要项目知识 + 判断
写代码AI（developer / dev-*）核心智能工作
审查代码AI（code-reviewer）需要语义比对
路由任务到合适 agentAI（pm-orchestrator）需要读 Architect 的 partition 决策
同步 .harness/agents 和 .harness/skills 到 .claude/代码（harness-sync）机械 byte 复制，每次结果应当一致。
注：v0.10 起规则不再合成进 CLAUDE.md，改由 AI-GUIDE.md 索引 + AI 按需 lazy-load。
检查 .harness/ 与 .claude/ 一致代码（harness-sync --check）byte 比对，AI 容易看走眼
检查仓库结构完整性代码（verify_all）CI 友好；秒级执行
跑测试 / 编译 / lint代码 + 项目工具链已有工具，AI 不需要重做
模板复制 + 占位符替换代码（test-init 模拟，真实 init 走 Skill）确定性 ↑，速度 ↑

"全 AI" 的代价

✅ 全 AI 的好处

  • 代码量极少：几个 Skill md 文件即可
  • AI 升级自动跟进，不用维护脚本
  • 跨平台无问题（不依赖 PS / Bash）
  • 灵活，AI 根据具体项目调整

❌ 全 AI 的代价

  • 失去 CI 集成：CI 跑不了 Claude Code，verify 必须在会话内做
  • 失去可重复性：同一输入不同会话可能微妙差异，无法 byte 比对
  • 失去自动化回归测试：test-init / test-real-project 这种"代码验证代码"的护栏没了
  • 速度慢：脚本 1ms，AI 调用秒级
  • 成本：每次操作消耗 token
  • 调试困难：脚本 bug 用 print 找，AI 行为 bug 找半天
  • 固化的工程知识丢了：脚本是"组织的肌肉记忆"，删了就没了

设计原则：代码用于"机制层"，AI 用于"内容层"

💡 核心区分

机制（代码）：怎么把文件从 A 拷到 B，怎么判断两份是否一致，怎么跑测试。
内容（AI）：写什么需求，设计什么方案，怎么实现，怎么审查。
机制类工作有"确定正确答案"，应该用代码；内容类工作需要判断，应该用 AI。混着用会丢两边的优点。

OpenAI Harness Engineering 文章也是这思路

参考资料里 OpenAI 的 Codex Harness 报告（100 万行代码 / 1500 PR / 零手写）也是"代码 harness + AI 执行"。 他们的 harness 里有大量脚本（linter、CI 检查、validators）。AI 在脚本框架内工作。 关键引用："Humans steer. Agents execute." 这套设计跟我们的思路一致。

哪些部分以后可以转 AI（v0.5+ 候选）

候选现状转 AI 后权衡
harness-sync 的 rules 合成逻辑 脚本按文件名排序 + cat AI 读 rules/ 再写 CLAUDE.md 当前实现已经简单，转 AI 收益小
模板占位符替换 {{PROJECT_NAME}} → 用户回答，机械替换 AI 直接根据用户回答生成定制内容（不需要模板） 失去模板可审计性；得到更灵活的定制
项目类型侦察（adopt 用） 检查 package.json / pyproject.toml 等 AI 直接读项目结构判断 AI 已经在做这个，代码版只是辅助；可考虑全删
规则提取（adopt 用） 关键字 match README / CONTRIBUTING AI 语义提取（更准） 转 AI 明显改善质量
verify_all 主体 脚本检查文件存在/同步等 —（不建议转） CI 集成必要 + 可重复性必要
test-init / test-real-project 脚本断言 —（不建议转） 自动化回归必须代码化

实际的演化路径

  1. 当前 v0.4.x：代码做机制，AI 做内容。基础打稳。
  2. v0.5 候选：规则提取转 AI（adopt 的弱点），其他保持。同时做 backend 分区。
  3. v0.6+ 看反馈：如果维护代码确实开销大、AI 越来越可靠，逐步把"代码做机制"中可转的转给 AI。但永远保留 verify_all + 自动化回归（这是工程底线）。

给你的决策框架

💡 什么时候把"代码"转成"AI"

问这 4 个问题：

  1. 这件事 需要 CI 跑 吗？是 → 保留代码。否 → 可转 AI。
  2. 这件事 需要 byte-identical 可重复 吗？是 → 保留代码。否 → 可转 AI。
  3. 这件事 有判断 / 语义理解 成分吗？是 → 转 AI 受益。否 → 代码更好。
  4. 这件事 频率高 吗（每个 commit / 每天跑多次）？是 → 代码（成本 + 速度）。否 → AI 也行。

3 票"转 AI"才转。当前每个保留为代码的部分都有 ≥3 票"保留"。
