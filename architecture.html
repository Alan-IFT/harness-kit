v2 · Claude Code 专用精简版

核心思路：Claude Code 已经提供机制层（Sandbox、Hooks、子 Agent、MCP、Memory、Skills 等），我们只补"项目级应用资产"。剔除个人项目用不上的复杂模块。

💡 关键认知：Claude Code 提供"机制"，你写"内容"
Claude Code 给你的是基础设施（hooks 机制、sub-agent 机制、permission 机制、MCP 机制），但每个机制具体做什么、什么内容、什么提示词，都是项目特定的，必须自己写。

v2 已从方案中移除的模块（根据你的反馈）

❌ Vault / 凭证管理
个人项目用不上，移除。凭证用环境变量 + Claude Code permission 拦截就够。
❌ 复杂 Token Budget 分级
Claude Code 自带 auto-compaction + /cost + 模型选择，个人项目够用。
保留极简策略：给 Developer Agent 配最强模型，其他用默认。
❌ 重型 4 层 Eval Pipeline
简化为轻量回归：留 2-3 个金标准任务在 evals/ 目录，每次大改 Rule/Skill 后手跑一遍。
❌ 生产级 Trace / 可观测系统
简化为：Claude Code session transcript + /cost 就够。等团队规模化才补 Langfuse / OpenTelemetry。
❌ 多平台同步
只支持 Claude Code，移除 .cursor/ .github/copilot-instructions.md 同步逻辑。
❌ 重型 Tool Registry
用 Claude Code settings.json 的 allow/deny lists 替代。需要 RBAC/审计的企业场景再说。

v2 落地路径（个人项目）

🚀 Phase 1 · MVP 1 周

  • 跑 Bootstrap 生成 .claude/ 骨架
  • 手写 SPEC + CLAUDE.md 最小规则（≤5 条）
  • 写 verify_all 基础检查项（编译+测试+基线）

🔧 Phase 2 · Hardening 2~3 周

  • 补全 Skills 内容（按项目栈定制）
  • 填充 dev-map + 任务看板
  • 加 hooks（如：保存后自动 lint）
  • 建 evals/ 金标准任务集
  • 需要外部能力时接 MCP（如 GitHub / 内部 Wiki）

🌐 Phase 3 · 可选扩展 按需

  • 团队上规模时再考虑 Vault
  • 跨项目 Tool Registry
  • Langfuse 等 Trace 系统

个人项目大多止步 Phase 2
v2 心态：把 Claude Code 当成"开箱即用的 Harness 运行时"，你只补"应用层资产"。

概念解释：4 层 Eval、Trace、Token Budget 到底是什么

这里把第三篇/第四篇文章里那些"听起来很重"的概念拆开讲，并对照 Claude Code 的实际能力，告诉你"个人项目到底需不需要"。

① Token Budget — Claude Code 自带到什么程度？

核心问题：Multi-Agent 容易烧钱（每个 Agent 都有 System Prompt，工具结果会塞回模型，重试历史不断膨胀），所以需要"预算管理"。

❌ 生产级版本（文章4，过重）
  • 按任务/Agent 分配 Token 配额
  • 剩余预算分级降级（绿/黄/红/熔断）
  • 不同档位模型自动路由
  • 单位业务结果成本统计看板
  • 预算熔断次数监控
✅ Claude Code 自带 + v2 极简
  • 自带：上下文 auto-compaction（接近窗口自动压缩）
  • 自带：/cost 查看本次会话累计 Token
  • 自带：Anthropic API 速率限制
  • v2 极简：Developer Agent 配 Opus，其他配 Sonnet/Haiku
  • v2 极简：CLAUDE.md 里写"重试 ≥3 次必须停"
结论：个人项目用 Claude Code 自带 + 按角色配模型，不需要复杂分级降级。

② 4 层 Eval 评估体系 — 它到底解决什么问题？

核心问题：传统软件改代码会跑单元测试防止退化；AI 系统改 Prompt / 改 Rule / 换模型也需要"回归测试"。

四层是：

层评估什么方法
L1 组件单 Agent 输出是否符合契约JSON Schema 校验
L2 轨迹工作流程是否有重复/循环/无效步骤分析 trace（关键创新）
L3 任务完成度最终结果是否达成用户目标LLM-as-Judge + 人工抽检
L4 端到端业务指标（采纳率/返工率/成本）线上反馈
❌ 生产级版本（过重）
  • 每次改 Prompt 跑 CI 回归
  • 建 Eval 数据集（几十~几百条）
  • LLM-as-Judge + 规则校验 + 人工抽检三合一
  • A/B 测试不同模型/Prompt
  • 采纳率/返工率/成本看板
✅ v2 极简轻量回归
  • 在 evals/golden-tasks.md 留 2-3 个金标准任务
  • 大改 Rule / Skill / Agent 提示词后手跑一遍
  • 看是否还能正确走完 7-Agent 流水线
  • 看 verify_all 是否通过
  • 个人项目这就够了
结论：个人项目用"2-3 个金标准任务手跑"代替，不需要建 Eval Pipeline + CI。

③ Trace / 可观测性 — 这是什么，Claude Code 给到哪一步？

核心问题：Agent 出错往往不是"异常崩溃"，而是"过程偏移"——选错工具、读错记忆、误解目标、压缩丢约束。需要能"事后回看每一步"。

❌ 生产级 Trace 系统（过重）
  • OpenTelemetry / Langfuse 接入
  • 每次 LLM 调用 + 工具调用打 span
  • 跨任务统计看板（成功率/重试率/熔断率）
  • 失败模式聚类分析
  • Session 不可变事件流支持回放
  • 跨多个 Agent 的因果链追踪
✅ Claude Code 自带 + v2 极简
  • 自带：session transcript（对话记录）
  • 自带：/cost 用量
  • 自带：每次工具调用在对话里就能看到
  • v2 加：verify_all 每次跑追加到 verification_history.log
  • v2 加：每个任务 6 阶段文档天然就是 trace
结论：个人项目看 transcript + 文档链就够。不需要接 Langfuse。

项目类型模板（仅支持全栈 / 后端）

A · 全栈（前端 + 后端 + DB）
规则示例：API 契约 / 前后端类型同步 / 数据库迁移规范 / 环境变量管理 / 错误处理统一
verify_all 检查项：
  • 前端 lint + 类型检查（tsc / eslint）
  • 后端 lint + 单测
  • OpenAPI / tRPC schema 一致性
  • 数据库迁移可回滚
  • E2E 烟雾测试（Playwright 关键路径）
  • bundle size / API 响应时间基线
典型技术栈：Next.js / Nuxt / Remix · NestJS / FastAPI / Express · Prisma / Drizzle / TypeORM
B · 纯后端 / API 服务
规则示例：API 契约 / 错误码规范 / 日志结构化 / 数据库事务边界 / 幂等性 / 速率限制
verify_all 检查项：
  • lint + 类型检查
  • 单元测试 + 集成测试
  • OpenAPI 校验
  • SQL 慢查询扫描
  • 安全扫描（依赖漏洞 / 密钥泄露）
  • 测试覆盖率基线只升不降
典型技术栈：Node/TS (NestJS / Fastify) · Python (FastAPI / Django) · Go (Gin / Echo) · Rust (Axum)
📌 暂不支持的项目类型
WPF / Unity / 纯前端 / 数据 ML 流水线等，按你的需求先不做。后续如果项目类型扩展，加新模板即可。

v2 + Bootstrap 总结建议

  1. 采用 v2 Claude Code 精简版 — 不重造轮子，专注项目层资产
  2. Bootstrap 用 Claude Code Skill 形态（/harness-init + /harness-adopt）
  3. 从 Phase 1 MVP 开始，1 周跑通一条端到端链路
  4. 个人项目止于 Phase 2；Phase 3 留给团队规模化时按需补
  5. 剔除模块按需还能再加回来（Vault / Eval / Trace 不是永远不做，是先不做）
