# Upgrading to the `.harness/scripts/` layout (T-007)

Harness now keeps all of its own scripts under **`.harness/scripts/`** instead of a
top-level `scripts/` directory, so harness-owned scripts no longer share a folder
with your project's own scripts. Fresh `/harness-init` projects are already on the
new layout — nothing to do.

**If your project was initialized before this change** (scripts under `scripts/`),
run the one-shot helper once from your project root:

```bash
# Windows
pwsh -NoProfile -File .harness/scripts/migrate-scripts-layout.ps1
# macOS / Linux
bash .harness/scripts/migrate-scripts-layout.sh
```

(Don't have the helper yet? It ships under `.harness/scripts/` in this release; if
your tree predates it, copy `migrate-scripts-layout.{ps1,sh}` from the harness-kit
templates' `common/.harness/scripts/` first.)

The helper:

- moves the known harness-owned scripts (`verify_all`, `harness-sync`, `guard-rm`,
  `install-hooks`, `archive-task` pairs + `baseline.json`) from `scripts/` to
  `.harness/scripts/` — your own `scripts/<custom>` files are left untouched;
- rewires the two hook command strings in `.claude/settings.json` (Stop +
  PreToolUse) to the new path, after writing a timestamped `.bak` backup;
- is **idempotent** — safe to run twice; a second run is a clean no-op.

Pass `-DryRun` / `--dry-run` to preview, or `-Force` / `--force` to overwrite an
existing target. After migrating, run `.harness/scripts/verify_all` to confirm the
gate is green. The old `scripts/verify_all` is gone — invoke `.harness/scripts/verify_all`.

# Migration Guide: v0.1.x → v0.2.0

If you have a project that was initialized with `harness-init` v0.1.x and want to
upgrade it to v0.2 (tool-agnostic `.harness/` layer + binding sync), this guide
walks you through the manual migration.

> An automated upgrade Skill is planned for v0.3 (`/harness-upgrade`). For now,
> the steps below take 5–10 minutes per project.

## Before you start

```bash
# 1. Make sure you're on a clean working tree
git status

# 2. Commit anything in progress
git add -A && git commit -m "wip: pre-upgrade snapshot"

# 3. Create an upgrade branch (optional but recommended)
git checkout -b chore/upgrade-harness-0.2
```

## Concept summary

v0.1 layout:
```
.claude/
  agents/*.md     ← you edit this directly
  skills/*/SKILL.md
  settings.json
CLAUDE.md         ← you edit this directly
```

v0.2 layout:
```
.harness/         ← edit here
  agents/*.md
  rules/*.md
  skills/*/SKILL.md
.claude/          ← generated; do not edit
  agents/*.md
  skills/*/SKILL.md
  settings.json
CLAUDE.md         ← generated; do not edit
scripts/
  harness-sync.{ps1,sh}   ← new
```

## Step 1: Bring in the new scripts

Update your Harness Engineering installation first:

```powershell
cd ~/harness-kit
git pull
& ./install.ps1     # refresh the skills
```

```bash
cd ~/harness-kit
git pull
./install.sh
```

Then in your project, copy the binding sync script from the templates:

```powershell
# Windows
Copy-Item ~/harness-kit/skills/harness-init/templates/common/scripts/harness-sync.ps1 scripts/
Copy-Item ~/harness-kit/skills/harness-init/templates/common/scripts/harness-sync.sh scripts/
```

```bash
cp ~/harness-kit/skills/harness-init/templates/common/scripts/harness-sync.ps1 scripts/
cp ~/harness-kit/skills/harness-init/templates/common/scripts/harness-sync.sh scripts/
```

## Step 2: Move agents to `.harness/agents/`

```powershell
# Windows
New-Item -ItemType Directory -Path .harness/agents -Force
Move-Item .claude/agents/*.md .harness/agents/
```

```bash
mkdir -p .harness/agents
mv .claude/agents/*.md .harness/agents/
```

## Step 3: Move skills to `.harness/skills/`

```powershell
# Windows
New-Item -ItemType Directory -Path .harness/skills -Force
if (Test-Path .claude/skills/build)  { Move-Item .claude/skills/build  .harness/skills/ }
if (Test-Path .claude/skills/test)   { Move-Item .claude/skills/test   .harness/skills/ }
if (Test-Path .claude/skills/verify) { Move-Item .claude/skills/verify .harness/skills/ }
# If you have other custom skills, move them too.
```

```bash
mkdir -p .harness/skills
for s in build test verify; do
    [[ -d .claude/skills/$s ]] && mv .claude/skills/$s .harness/skills/
done
# Move any custom skills as well.
```

## Step 4: Split CLAUDE.md into rule fragments

This is the only step that needs a little judgment. Your existing `CLAUDE.md` becomes
one or more files under `.harness/rules/`.

The simplest path is one big file:

```powershell
New-Item -ItemType Directory -Path .harness/rules -Force
Move-Item CLAUDE.md .harness/rules/00-core.md
```

```bash
mkdir -p .harness/rules
mv CLAUDE.md .harness/rules/00-core.md
```

If your CLAUDE.md was originally generated by harness-init v0.1 with a fullstack or
backend overlay, you'll see a `## Fullstack-specific rules` (or backend) section that
came from `.append`. Split it out:

```bash
# Find the marker line and split there. Example with awk:
awk '/^## (Fullstack|Backend)-specific rules$/ {f=1} !f {print}' .harness/rules/00-core.md > /tmp/00-core-base.md
awk '/^## (Fullstack|Backend)-specific rules$/ {f=1} f {print}' .harness/rules/00-core.md > .harness/rules/50-overlay.md
mv /tmp/00-core-base.md .harness/rules/00-core.md
```

After splitting, rename `50-overlay.md` to `50-fullstack.md` or `50-backend.md`
depending on your project type.

You can also keep everything in `00-core.md` if you don't want to split. The
fragment mechanism doesn't *require* multiple files; it just supports them.

## Step 5: Run the binding sync

```powershell
pwsh -File scripts/harness-sync.ps1
```

```bash
bash scripts/harness-sync.sh
```

This regenerates `.claude/agents/`, `.claude/skills/`, and `CLAUDE.md` from
`.harness/`. Expected output: `Sync complete (N item(s) updated)`.

## Step 6: Verify

```powershell
pwsh -File scripts/verify_all.ps1
```

```bash
bash scripts/verify_all.sh
```

If your generated-project verify_all is the v0.1.x version, it doesn't have the
`E.4 Binding in sync` check yet. To upgrade verify_all itself, regenerate from the
new template:

```powershell
# Replace scripts/verify_all.* with the v0.2 templates after substituting placeholders.
# Simplest: re-run harness-init in a sibling dir with your same type/stack, copy over.
```

Or just inspect the new template at
`~/harness-kit/skills/harness-init/templates/<type>/scripts/verify_all.{ps1,sh}.tmpl`
and merge the new check section into your existing script.

## Step 7: Commit

```bash
git add -A
git commit -m "chore: upgrade Harness to v0.2 (tool-agnostic .harness/ layer)"
```

## What changed in behavior

| Action | Before (v0.1) | After (v0.2) | Refinement (v0.10+) |
|---|---|---|---|
| Edit an agent | Edit `.claude/agents/foo.md` directly | Edit `.harness/agents/foo.md`, run `harness-sync` | Unchanged |
| Edit a project rule | Edit `CLAUDE.md` directly | Edit a `.harness/rules/NN-*.md` fragment, run `harness-sync` | Edit the fragment; **no sync needed** — rules are referenced by `AI-GUIDE.md`, not composed |
| Add a project skill | `mkdir .claude/skills/foo`, write SKILL.md | `mkdir .harness/skills/foo`, write SKILL.md, run `harness-sync` | Unchanged |
| Edit settings.json | Direct edit | Direct edit (still in `.claude/`, not part of sync) | Unchanged |
| Verify project | `verify_all` | `verify_all` (now also checks E.4 Binding in sync) | `verify_all` now has 29 checks (E.2 byte-identity + E.4b AI-GUIDE ↔ rules drift + I.* doc-size + G.3 version stamps + F.2 guard-rm + I.6 retired-claim guard + D.3 AI-generated 50-*.md sanity) |

## Troubleshooting

**`harness-sync` reports drift right after I ran it.**
You may have orphans: files in `.claude/agents/` that aren't in `.harness/agents/`.
The sync removes them on the second run. Run `harness-sync` again.

**My CLAUDE.md edits keep disappearing.** *(v0.2–v0.9 behavior)*
You're editing the file that used to be regenerated by `harness-sync`. Edit
`.harness/rules/*.md` instead, then sync. *Since v0.10, `CLAUDE.md` is a static
stub written once at init and is no longer regenerated by `harness-sync` — your
edits will persist, but the right place to encode rules is still
`.harness/rules/*.md` (AI tools follow the pointer chain CLAUDE.md → AI-GUIDE.md →
rule fragments).*

**verify_all step E.4 doesn't exist for my project.**
Your project's `scripts/verify_all.*` is the v0.1.x version. See Step 6 above to
update it.

**I accidentally deleted `.harness/` and only have `.claude/` now.**
You can reverse the migration. Move agents and skills back from `.claude/` to
`.harness/` (Steps 2 and 3 in reverse), then run sync. The content is preserved
because the binding generator was a one-way copy.

## Rolling back

If something goes wrong:

```bash
git checkout main
git branch -D chore/upgrade-harness-0.2
```

You're back where you started.
