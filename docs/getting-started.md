# Getting Started

This guide walks you from "I just installed Harness Kit" to "I'm shipping features through it".

## Prerequisites

- [Claude Code](https://docs.claude.com/claude-code) installed and authenticated
- Git
- A project (or an empty folder if starting fresh)
- PowerShell 7+ (Windows) or Bash (macOS / Linux)

## 1. Install the skills

The recommended path is the Claude Code Plugin Marketplace (skills appear namespaced
as `/harness-kit:harness-init`, etc.):

```
/plugin marketplace add Alan-IFT/harness-kit
/plugin install harness-kit@harness-kit-marketplace
```

Or, for a global user-skill install (callable as `/harness-init` without the namespace):

```powershell
# Windows
git clone https://github.com/Alan-IFT/harness-kit ~/harness-kit
& ~/harness-kit/install.ps1
```

```bash
# macOS / Linux
git clone https://github.com/Alan-IFT/harness-kit ~/harness-kit
~/harness-kit/install.sh
```

Either path makes seventeen skills available in Claude Code:

Pipeline (pick by task shape; plus a pre-pipeline aligner):

- `harness-grill` тАФ pre-pipeline alignment interview (one question at a time, a recommended answer per question, self-answers from the codebase) that emits an aligned brief and stops; hand the brief to `harness` or a `harness-stream` pool
- `harness` тАФ full 7-stage pipeline (RA тЖТ SA тЖТ GR тЖТ Dev тЖТ CR тЖТ QA тЖТ Delivery)
- `harness-plan` тАФ design-only mode (stops at Gate Review verdict)
- `harness-explore` тАФ research / feasibility mode
- `harness-goal` тАФ open-ended Dev + QA loop bounded by a measurable criterion
- `harness-batch` тАФ run a fixed list of tasks through the pipeline sequentially (fail-stop)
- `harness-stream` тАФ drain a living task pool you keep topping up mid-run (best-effort)

Setup:

- `harness-init` тАФ bootstrap a new project
- `harness-adopt` тАФ non-invasively add Harness to an existing project
- `harness-upgrade` тАФ upgrade an already-initialized but stale project to the current plugin layout
- `harness-language` тАФ set / switch (English тЖФ Chinese) / refresh a project's output-language policy

Operations:

- `harness-verify` тАФ run total verification
- `harness-status` тАФ health snapshot
- `harness-intervene` тАФ soft Ctrl-C for an in-flight pipeline (redirect / pause / add-task)
- `harness-supervise` тАФ observer-only health check of a task folder
- `harness-decision-mode` тАФ switch the decision/escalation mode (Mode 1 human-decides / Mode 2 preset-rubric / Mode 3 custom-rubric)
- `harness-deflate` тАФ holistic anti-entropy sweep (v0.41+): scans the whole codebase via the read-only supervisor entropy lens, presents each finding with WHERE + a strength badge, and on your explicit authorization hands the chosen deepening to `/harness-goal` (machine reminds, you authorize, machine executes)

Verify with `/help` inside Claude Code тАФ you should see them listed.

## 2. Bootstrap your project

### New project (empty folder)

```
cd /path/to/empty/folder
claude
> /harness-init
```

Answer the six questions (project type, stack, verify hook, Developer partitioning, output language, and opt-in AI customization of the `50-<project>.md` rule fragment). The skill generates a
**two-layer structure**:

```
your-project/
тФЬтФАтФА .harness/                     тЖР SOURCE OF TRUTH (edit these)
тФВ   тФЬтФАтФА agents/                  тЖР 7 sub-agents
тФВ   тФЬтФАтФА rules/                   тЖР rule fragments (00-core, 50-overlay тАж)
тФВ   тФЬтФАтФА skills/                  тЖР build, test, verify procedures
тФВ   тФФтФАтФА scripts/                 тЖР verify_all, harness-sync, baseline (relocated here in v0.20)
тФЬтФАтФА .claude/                      тЖР Claude Code binding
тФВ   тФЬтФАтФА agents/                  тЖР regenerated by harness-sync from .harness/agents/
тФВ   тФЬтФАтФА skills/                  тЖР regenerated by harness-sync from .harness/skills/
тФВ   тФФтФАтФА settings.json            тЖР direct binding glue (edit if needed)
тФЬтФАтФА AI-GUIDE.md                   тЖР tool-agnostic entry; indexes .harness/rules/
тФЬтФАтФА CLAUDE.md                     тЖР ~15-line stub pointing at AI-GUIDE.md (written once at init)
тФЬтФАтФА docs/
тФВ   тФЬтФАтФА workflow.md              тЖР 7-stage pipeline definition
тФВ   тФЬтФАтФА spec/                    тЖР write your project SPECs here
тФВ   тФЬтФАтФА dev-map.md               тЖР navigate the codebase
тФВ   тФЬтФАтФА tasks.md                 тЖР project task board
тФВ   тФФтФАтФА features/                тЖР per-task documents
тФФтФАтФА evals/golden-tasks.md         тЖР regression set
```

### Existing project (already has code)

```
cd /path/to/existing/project
claude
> /harness-adopt
```

`/harness-adopt` detects your stack, extracts existing conventions, and prompts before
applying тАФ it does not silently overwrite existing files.

## 3. Your first task

In Claude Code, give the PM Orchestrator a task:

```
Take this task: Add a CSV export button to the orders list page.
```

The PM will:
1. Create `docs/features/csv-export-orders/`.
2. Read `docs/tasks.md` for related history.
3. Dispatch the Requirement Analyst.
4. Dispatch each subsequent stage in order, checking gate criteria.
5. At the end, write `07_DELIVERY.md` and update `docs/tasks.md`.

You typically only need to:
- Answer ambiguity questions when the analyst asks.
- Review at Gate (stage 3) and Code Review (stage 5).
- Confirm any production-risky action.

## 4. Editing Harness (the right way)

> **Rule of thumb**: edit `.harness/`, never `.claude/`. `CLAUDE.md` is a one-time stub
> written at init тАФ leave the AI-GUIDE.md pointer intact.

Since v0.10, rules are **not** composed into CLAUDE.md. AI tools follow the pointer
from CLAUDE.md тЖТ `AI-GUIDE.md` тЖТ `.harness/rules/*.md` and lazy-load only the fragments
whose "when to read" trigger matches the task. No regeneration needed.

### To change a project rule

```bash
# Edit a rule fragment (or add a new one with a higher NN prefix)
vim .harness/rules/00-core.md
# or: vim .harness/rules/80-my-team.md   (new file)

# If you added a new fragment, add a one-line index entry in AI-GUIDE.md
# (verify_all step E.4b enforces this).

# Verify
pwsh -File .harness/scripts/verify_all.ps1
```

No `harness-sync` needed for rule edits тАФ rules are referenced by AI-GUIDE.md, not copied.

### To change an agent's behavior

```bash
vim .harness/agents/developer.md
pwsh -File .harness/scripts/harness-sync.ps1     # syncs .harness/agents/ тЖТ .claude/agents/
pwsh -File .harness/scripts/verify_all.ps1
```

### To add a project-specific skill

```bash
mkdir -p .harness/skills/<name>
# Write .harness/skills/<name>/SKILL.md with proper Claude Code frontmatter
pwsh -File .harness/scripts/harness-sync.ps1     # syncs .harness/skills/ тЖТ .claude/skills/
```

After sync, the skill appears in `.claude/skills/<name>/SKILL.md` and is usable
in Claude Code on the next session.

### `verify_all` enforces it

If you edit `.harness/agents/` or `.harness/skills/` but forget to sync, the next
`/harness-verify` will FAIL on E.2 (`.claude/agents and .claude/skills synced from
.harness/`). Just run `harness-sync` and re-verify. The Stop hook in
`.claude/settings.json` runs `harness-sync` automatically at session end, so this
rarely bites you in Claude Code.

## 5. Daily workflow

| You want to... | Do this |
|---|---|
| Start a new feature or bug fix | Tell PM Orchestrator: "Take this task: ..." |
| Check verify is still green | `/harness-verify` |
| See current Harness health | `/harness-status` |
| Add / change a project-wide rule | Edit a file in `.harness/rules/`, run `harness-sync` |
| Add a repeatable operation | Add a skill under `.harness/skills/<name>/SKILL.md`, sync |
| Update what files exist where | Edit `docs/dev-map.md` (the developer agent does this for you) |
| Run regression after Harness changes | Re-run tasks in `evals/golden-tasks.md` |

## 6. When AI makes a mistake

The Harness philosophy: **don't retry; fix the rail**.

| The mistake | Where to fix it |
|---|---|
| AI used a forbidden API / style | Add a line in `.harness/rules/`; add a check in `verify_all` |
| AI forgot a step (e.g. lint before commit) | Add `.harness/skills/<name>/SKILL.md` |
| Two agents stepped on each other | Edit `.harness/agents/<agent>.md` to tighten the contract |
| AI lacks an external capability | Add an MCP server, register in `.claude/settings.json` |
| A whole stage of work was missed | Add a new agent and update `docs/workflow.md` |

Edit `.harness/`, run `harness-sync`, verify. Mistakes ratchet up the Harness instead
of repeating.

## 7. What you shouldn't expect this to do

- **Replace your testing**. It verifies tests run; you still write meaningful ones.
- **Replace code review for security-critical code**. Use `/security-review` or human review.
- **Decide whether to ship**. The pipeline produces a verdict; you decide.
- **Operate production**. Anything destructive is HITL by design.

## Troubleshooting

**Skill not found in Claude Code.**
Check `~/.claude/skills/harness-init/SKILL.md` exists. Restart Claude Code if you installed it mid-session.

**verify_all says PowerShell can't run scripts.**
On Windows you may need `Set-ExecutionPolicy -Scope CurrentUser RemoteSigned`.

**verify_all step E.2 fails: ".claude/agents and .claude/skills synced from .harness/".**
You edited `.harness/agents/` or `.harness/skills/` but forgot to sync. Run
`.harness/scripts/harness-sync.{ps1,sh}` and re-verify.

**The PM keeps rolling back to the analyst.**
Probably an ambiguous requirement. Read the analyst's questions and answer them тАФ the loop ends when nothing is ambiguous.

**My `.claude/agents/foo.md` edits disappeared.**
You're editing the generated copy. Edit `.harness/agents/foo.md` instead, then run `harness-sync`. Future edits stick.

**I want the AI to follow a different rule.**
Edit (or add) a fragment in `.harness/rules/`, and make sure `AI-GUIDE.md` indexes it
with a "when to read" trigger. Use `NN-name.md` where `NN` is a 2-digit prefix: 00-19
core, 20-49 cross-cutting, 50-79 project-type overlays, 80+ project-specific.
