# Getting Started

This guide walks you from "I just installed Harness Kit" to "I'm shipping features through it".

## Prerequisites

- [Claude Code](https://docs.claude.com/claude-code) installed and authenticated
- Git
- A project (or an empty folder if starting fresh)
- PowerShell 7+ (Windows) or Bash (macOS / Linux)

## 1. Install the skills

```powershell
# Windows
git clone https://github.com/<your>/harness-kit ~/harness-kit
& ~/harness-kit/install.ps1
```

```bash
# macOS / Linux
git clone https://github.com/<your>/harness-kit ~/harness-kit
~/harness-kit/install.sh
```

This drops four skills into `~/.claude/skills/`:

- `harness-init`
- `harness-adopt`
- `harness-verify`
- `harness-status`

Verify with `/help` inside Claude Code — you should see them listed.

## 2. Bootstrap your project

### New project (empty folder)

```
cd /path/to/empty/folder
claude
> /harness-init
```

Answer the three questions (type, stack, verify hook). The skill generates a
**two-layer structure**:

```
your-project/
├── .harness/                     ← SOURCE OF TRUTH (edit these)
│   ├── agents/                  ← 7 sub-agents
│   ├── rules/                   ← rule fragments (00-core, 50-overlay …)
│   └── skills/                  ← build, test, verify procedures
├── .claude/                      ← GENERATED (do not edit; regenerated by harness-sync)
│   ├── agents/
│   ├── skills/
│   └── settings.json            ← (this one is direct binding glue, edit if needed)
├── CLAUDE.md                     ← GENERATED (do not edit)
├── docs/
│   ├── workflow.md              ← 7-stage pipeline definition
│   ├── spec/                    ← write your project SPECs here
│   ├── dev-map.md               ← navigate the codebase
│   ├── tasks.md                 ← project task board
│   └── features/                ← per-task documents
├── scripts/
│   ├── verify_all.{ps1,sh}      ← total verification
│   ├── harness-sync.{ps1,sh}    ← regenerates .claude/ + CLAUDE.md from .harness/
│   └── baseline.json            ← test/quality baseline (only goes up)
└── evals/golden-tasks.md         ← regression set
```

### Existing project (already has code)

```
cd /path/to/existing/project
claude
> /harness-adopt
```

> **Note**: in v0.1.x/0.2.x, adopt is *scaffolding-only* — it scans your repo and writes a
> plan to `.harness-adopt/PLAN.md`. You then apply manually. Full automated adopt
> ships in v0.3.

## 3. Your first task

In Claude Code, give the PM Orchestrator a task:

```
Take this task: Add a CSV export button to the orders list page.
```

The PM will:
1. Create `docs/features/csv-export-orders/`.
2. Read `docs/tasks.md` for related history.
3. Dispatch the Requirement Analyst.
4. Dispatch each subsequent stage in order, checking gate criteria.
5. At the end, write `07_DELIVERY.md` and update `docs/tasks.md`.

You typically only need to:
- Answer ambiguity questions when the analyst asks.
- Review at Gate (stage 3) and Code Review (stage 5).
- Confirm any production-risky action.

## 4. Editing Harness (the right way)

> **Rule of thumb**: edit `.harness/`, never `.claude/` or `CLAUDE.md`.

### To change a project rule

```bash
# Edit a rule fragment (or add a new one with a higher NN prefix)
vim .harness/rules/00-core.md
# or: vim .harness/rules/80-my-team.md   (new file)

# Regenerate CLAUDE.md
pwsh -File scripts/harness-sync.ps1     # Windows
bash scripts/harness-sync.sh            # Unix

# Verify
pwsh -File scripts/verify_all.ps1
```

### To change an agent's behavior

```bash
vim .harness/agents/developer.md
pwsh -File scripts/harness-sync.ps1
pwsh -File scripts/verify_all.ps1
```

### To add a project-specific skill

```bash
mkdir -p .harness/skills/<name>
# Write .harness/skills/<name>/SKILL.md with proper Claude Code frontmatter
pwsh -File scripts/harness-sync.ps1
```

After sync, the skill appears in `.claude/skills/<name>/SKILL.md` and is usable
in Claude Code on the next session.

### `verify_all` enforces it

If you ever forget to sync, the next `/harness-verify` (or `pwsh scripts/verify_all.ps1`)
will FAIL on E.4 (`Binding in sync`). Just run `harness-sync` and re-verify.

## 5. Daily workflow

| You want to... | Do this |
|---|---|
| Start a new feature or bug fix | Tell PM Orchestrator: "Take this task: ..." |
| Check verify is still green | `/harness-verify` |
| See current Harness health | `/harness-status` |
| Add / change a project-wide rule | Edit a file in `.harness/rules/`, run `harness-sync` |
| Add a repeatable operation | Add a skill under `.harness/skills/<name>/SKILL.md`, sync |
| Update what files exist where | Edit `docs/dev-map.md` (the developer agent does this for you) |
| Run regression after Harness changes | Re-run tasks in `evals/golden-tasks.md` |

## 6. When AI makes a mistake

The Harness philosophy: **don't retry; fix the rail**.

| The mistake | Where to fix it |
|---|---|
| AI used a forbidden API / style | Add a line in `.harness/rules/`; add a check in `verify_all` |
| AI forgot a step (e.g. lint before commit) | Add `.harness/skills/<name>/SKILL.md` |
| Two agents stepped on each other | Edit `.harness/agents/<agent>.md` to tighten the contract |
| AI lacks an external capability | Add an MCP server, register in `.claude/settings.json` |
| A whole stage of work was missed | Add a new agent and update `docs/workflow.md` |

Edit `.harness/`, run `harness-sync`, verify. Mistakes ratchet up the Harness instead
of repeating.

## 7. What you shouldn't expect this to do

- **Replace your testing**. It verifies tests run; you still write meaningful ones.
- **Replace code review for security-critical code**. Use `/security-review` or human review.
- **Decide whether to ship**. The pipeline produces a verdict; you decide.
- **Operate production**. Anything destructive is HITL by design.

## Troubleshooting

**Skill not found in Claude Code.**
Check `~/.claude/skills/harness-init/SKILL.md` exists. Restart Claude Code if you installed it mid-session.

**verify_all says PowerShell can't run scripts.**
On Windows you may need `Set-ExecutionPolicy -Scope CurrentUser RemoteSigned`.

**verify_all step E.4 fails: "Binding drift".**
You edited `.harness/` but forgot to sync. Run `scripts/harness-sync.{ps1,sh}` and re-verify.

**The PM keeps rolling back to the analyst.**
Probably an ambiguous requirement. Read the analyst's questions and answer them — the loop ends when nothing is ambiguous.

**My `.claude/agents/foo.md` edits disappeared.**
You're editing the generated copy. Edit `.harness/agents/foo.md` instead, then run `harness-sync`. Future edits stick.

**Generated CLAUDE.md doesn't match my style.**
Edit a fragment in `.harness/rules/`. The composition is by filename order (00 first, 99 last). Add your overrides as a higher-numbered fragment.
