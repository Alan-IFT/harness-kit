harness-kit v0.11.0 改动详解

一句话： v0.11 在不引入新代码的前提下，从 lsdefine/GenericAgent 借了三种纪律：把"一刀切的 7-stage"拆成 4 种适配不同任务形态的模式；让 QA 必须像对手一样独立找反例而不是顺着 Dev 的测试读；引入 ≤30 行的跨任务记忆文件解决"每次任务都重学一遍"。
人工负担： 几乎为零，反而少了 ——AI 现在能根据你的自然语言自动挑模式，不用你记 skill 名。

1. v0.10 → v0.11 的整体架构对比

v0.10（改动前）

你提需求 → /harness（PM Orchestrator）→ 7 个 agent 顺序跑（PM 派活） 1. Requirement Analyst → 01_REQUIREMENT_ANALYSIS.md 2. Solution Architect → 02_SOLUTION_DESIGN.md 3. Gate Reviewer → 03_GATE_REVIEW.md（通过才往下） 4. Developer (or 分区) → 04_DEVELOPMENT.md 5. Code Reviewer → 05_CODE_REVIEW.md 6. QA Tester → 06_TEST_REPORT.md（顺着 Dev 的测试读） 7. → 07_DELIVERY.md + verify_all PASS 跨任务记忆：无（每次从零开始）

v0.11（改动后）

你提需求 → AI 看你说的话挑模式 → 跑对应分支 ┌─ /harness ── 完整 7-stage（同 v0.10） QA 阶段升级 ├─ /harness-plan ── 只跑 1-3（RA→SA→GR） 新增 ├─ /harness-explore ── 只跑轻量 1（RA + findings） 新增 └─ /harness-goal ── Dev+QA 循环 + 预算 新增 每条任务开始前 AI 读 .harness/insight-index.md（≤30 行跨任务事实） 每条任务结束后 AI 写 ## Insight 段（如果有），归档脚本自动整理
本质差别： v0.10 是"一条河流，所有水都得走"。v0.11 是"四条岔路 + 一个共享水库（insight）"——按任务形态走对应那条；常识用水库浇灌不用每条河重打一遍。

2. 改动 A：三个执行模式（Plan / Explore / Goal）

2.1 为什么需要

v0.10 只有一条 7-stage 流水线。但现实开发里很多任务不适合走完整流程：

真实场景v0.10 跑完整 7-stage 的问题v0.11 用什么模式
"想确认这库的 API 限流能不能扛住我们的量"仪式过重：写 RA + 评审 + 设计 + 写代码 + 测试 = 浪费 80% 时间/harness-explore
"我有 3 个架构方案想让你评审一下，先不动手"v0.10 没有"只评审不动手"模式；Dev 阶段会强行写代码/harness-plan
"持续重构这个模块直到 verify_all 全绿"v0.10 把它当一次性任务，跑完一轮就结束；不会循环/harness-goal
"在订单页加一个 CSV 导出按钮"v0.10 走完整流程，正合适/harness（不变）

2.2 四个模式各跑哪几个阶段

RA SA GR Dev CR QA 交付 ↓ ↓ ↓ ↓ ↓ ↓ ↓ 完整 /harness ✅ ✅ ✅ ✅ ✅ ✅ ✅ 100% 工作量 /harness-plan ✅ ✅ ✅ ❌ ❌ ❌ 📝 ~35% /harness-explore 轻量 ❌ ❌ ❌ ❌ ❌ 📄 ~15% /harness-goal ❌ ❌ ❌ 循环 ❌ 最终 ✅ 按预算定

2.3 决策树（AI 会按你说的话自动挑）

你的话听起来是 ──→ ├─ "改个 typo / 加个注释" → 直接编辑 + verify_all（不进流程） ├─ "能不能 ..." "这库支持吗" "可行不" → /harness-explore ├─ "评审下这个设计" "先别动手" → /harness-plan ├─ "持续优化直到 X" "重构到绿" → /harness-goal └─ 其他正经任务 → /harness（完整 7-stage）
📋 具体到三个模式的契约

/harness-plan — 只跑设计

  • 输入：一个 idea 或需求描述
  • 输出：01_REQUIREMENT_ANALYSIS.md + 02_SOLUTION_DESIGN.md + 03_GATE_REVIEW.md（带判决：APPROVED / CHANGES REQUIRED / REJECTED）
  • 不做：不写代码，不跑测试
  • 可恢复：判决 APPROVED 后，跑 /harness 同一个 task slug 会跳过 1-3 直接从 Dev 开始

/harness-explore — 调研

  • 输入：一个问题（"能不能..."）
  • 输出：轻量 01_REQUIREMENT_ANALYSIS.md（1 页）+ findings.md（带 source 引用 + 推荐 next step）
  • 不做：不做架构设计，不写实现代码
  • 可能产出：如果调研发现非显然的事实，AI 会建议你加进 insight-index

/harness-goal — 目标循环

  • 输入：① 一句话目标（"覆盖率到 80%"）② 可量化判据（"pytest --cov 输出 >= 80"）③ 预算（max iterations 或 max minutes）
  • 循环逻辑：Dev 改一轮 → verify_all 跑 → 量目标判据 → 没到再改一轮（最多 N 轮或 M 分钟）
  • 每轮的 context 控制：Dev 只看最近 3 轮历史（不让 context 滚雪球）
  • 结束时：QA 跑一次最终验证（强制对抗性测试，见改动 B）
  • verdict：GOAL ACHIEVED 或 BUDGET EXHAUSTED，永不偷偷加预算

2.4 好处对比

任务v0.10 跑完整流程要多久v0.11 用对应模式节省
调研一个库~30 分钟（7 stage 都走）~5 分钟（explore）83%
架构评审~20 分钟（强制走到 QA）~8 分钟（plan）60%
"重构到绿"~跑一次 7-stage 后结束，绿了再跑一次循环到底，一次性体验质变
真实功能~30 分钟~30 分钟（不变）0%（正确）

3. 改动 B：对抗性验证（最高 ROI 的改动）

3.1 v0.10 QA 阶段的问题

问题： v0.10 的 QA Tester 是从 04_DEVELOPMENT.md 出发 ——读 Dev 写的测试代码、看 Dev 的报告、确认测试通过。

这导致： QA 继承了 Developer 的认知偏差。如果 Developer 写测试时假设了某个 mock 行为是对的（但其实是错的），QA 用同样的 mock 跑出来"通过"——bug 会过 QA 流出到生产。
真实例子： 开发者写 save(file)，写测试 expect(save('a.txt')).toBe(true)。这测试在 mock 文件系统下通过。但实际生产是 Windows，save 用了 / 路径分隔符 ——bug。QA 顺着读 Dev 的测试，看到"绿了"，放行。这就是共享假设杀人。

3.2 v0.11 怎么改

借鉴 GenericAgent 的 verify_sop.md 三铁律，把"对抗性"刻进 .harness/agents/qa-tester.md 的契约：

铁律含义
无 tool evidence = 无主张"看 diff 觉得对"不算验证。必须真跑代码，截 tool 输出贴到报告里。没跑过 = 没验证过。
独立 reproducer，不依赖 Dev 的测试QA 不能读 Dev 的测试代码后跑同样的东西。必须从 acceptance criterion 出发，自己写新的 reproducer。Dev 的测试只在 QA 自己的 reproducer 跑通后才被信任。
每条 AC 一个失败假设跑测试前，QA 必须先写："我预期这个会在 X 情况下挂掉，因为 <假设>。" 然后跑。挂了 → 加深理解 + 报 defect；没挂 → 写下你试过怎么破它、为什么破不掉。这强制对抗思维，不只是确认偏差。

3.3 06_TEST_REPORT.md 现在长什么样

v0.10（旧）

# Test Report
## Test plan
| AC | 测试 | 文件 |
| AC-1 user 能存档 | it('saves to disk') | tests/save.test.ts |

## verify_all 结果
Pass: 47 / Fail: 0

## Defects 发现
（无）

## Verdict
APPROVED FOR DELIVERY

问题：没看到 QA 自己测了什么，只看到 Dev 的测试通过了。

v0.11（新，REQUIRED 段加粗）

# Test Report
## Test plan
| AC | 测试 | 文件 |
| AC-1 user 能存档 | it('saves to disk') | tests/save.test.ts |

## Adversarial tests (REQUIRED)
| AC | 假设 | Reproducer | Outcome |
| AC-1 | 并发同名 save 会丢数据 | node tests/save-race.js（我写的）| 撑住了，输出：... |
| AC-2 | 输入 NUL 字节会崩 | pytest tests/test_nul.py（我写的）| **失败**，报 BLOCKER |

## verify_all 结果
Pass: 47 / Fail: 1 (新加的 NUL test)

## Defects 发现
[BLOCKER] save() 不处理 NUL 字节，崩在 fs.writeSync

## Verdict
CHANGES REQUIRED (1 BLOCKER)

关键：QA 写自己的测试找到了 Dev 没想到的 bug。

3.4 verify_all 怎么强制这件事

用户项目的 scripts/verify_all.{ps1,sh} 加了一条新 check：

// E.5 (fullstack) / D.5 (backend) — Adversarial tests section enforcement
扫描 docs/features/**/06_TEST_REPORT.md
对每个文件：grep -E '^##\s+Adversarial\s+tests'
如果有一个缺这段 → verify_all FAIL，输出 "Test reports missing Adversarial tests section: ..."

这意味着：QA 想偷懒不写对抗性测试 → verify_all 拦截 → 任务无法"declare done"。 contractually enforced，不靠人记。

4. 改动 C：Insight Index（跨任务记忆）

4.1 问题

v0.10 没有跨任务记忆。 每次新任务，PM Orchestrator + agent 都得从零理解项目。如果项目踩过一些非显然的坑（比如"这个 SDK 对非法 key 返回 null 不抛异常"、"WSL 下超过 4 个 watcher 会 ENOSPC"），下一个任务遇到同样情况会重新踩一遍。

项目跑 50 个任务之后，docs/features/ 下会有 50 × 7 = 350 个文档；想找"我们之前在哪个任务遇到过类似 bug"得手 grep；agent 不会主动去找。

4.2 v0.11 的解法（≤30 行的项目记忆文件）

新文件： .harness/insight-index.md —— 最多 30 行 的项目特有事实清单。每行一个事实 + 日期 + 证据指针。

新规则： .harness/rules/05-insight-index.md 定义契约 ——什么算 insight、什么时候读、什么时候写。

新脚本： scripts/archive-task.{ps1,sh} ——任务结束时自动从 07_DELIVERY.md 的 ## Insight 段收割 insight，移老的、归档任务文档。

4.3 insight-index.md 实际长什么样

# Insight Index — harness-kit > 跨任务踩坑事实。≤30 行。 > 设计/实现任务开始时读；只在证据支持的意外之后写。 - 2026-05-16 · Edit tool occasionally reports SUCCESS without applying the change — re-Read or Grep to verify. · evidence: v0.9.1 verify_all whitelist edit (commit 336e029) - 2026-05-16 · 任何新 {{...}} placeholder 必须加进 verify_all.ps1 AND .sh D.2 白名单 · evidence: v0.9.1 SYNC_COMMAND 加入时 - 2026-05-16 · sync-self 只同步 .harness/agents/ + 4 个脚本对，不同步 .harness/rules/ · evidence: v0.9.0 onward （最多 30 条，老的会被 archive-task 自动轮转走）

"Insight"的定义不是文档、不是 bug 报告、不是规则。是用证据击败合理先验的发现——"以为应该这样，结果不是；这里是证据。"

4.4 什么时候读、什么时候写

时机动作谁做
任务开始（设计/实现类）读 insight-index.md（30 行很快），检查有没有命中条目PM 或 Architect 或 Developer
任务执行中遇到"意外"在 07_DELIVERY.md 的 ## Insight 段写一行（必须带 evidence）Developer 或 QA
任务完成时跑 scripts/archive-task --task <slug> ——自动把 ## Insight 段追加到 insight-index.md + 归档任务文档PM
insight-index 超 30 行archive-task 自动把最老的轮转到 docs/features/_archived/insight-history.md脚本自动

4.5 写入纪律（防止污染）

05-insight-index.md 契约里写了 "对抗性写入测试"：

写之前问自己："一个有经验的人，新看代码库，能在 10 分钟内推导出这个吗？"
能 → 别写。那不是 insight，是普通文档/约定。
不能（必须踩过才知道）→ 写。

另外 4 类不能写进 insight-index：

  • Bug 报告（那在 docs/tasks.md）
  • 规则/约定（那在 .harness/rules/）
  • 最佳实践主张（代码/规则文件的位置）
  • 任务总结（那在 docs/features/<task>/）

7. 人工负担分析（最重要）

结论：人工负担基本为零，反而可能减少。

7.1 你不需要做的事

  • ❌ 不用记 4 个 skill 名 ——AI 看你自然语言提需求（"能不能..."、"先评审下..."、"持续优化..."）自动挑模式
  • ❌ 不用手动跑 archive-task ——PM Orchestrator 在任务交付时自动跑
  • ❌ 不用记得读 insight-index ——AI 按 AI-GUIDE.md 的契约自动在任务开始时读
  • ❌ 不用记得写 ## Adversarial tests 段 ——QA agent 的契约里写死了，verify_all 强制 enforce
  • ❌ 不用判断 insight 该不该写 ——AI 按 05-insight-index.md 的"对抗性写入测试"自检

7.2 你多了的事（可选，不强制）

事项频率负担
偶尔 review 一下 insight-index.md 的新条目，确认质量~每月一次扫一眼30 秒（30 行很快读完）
如果 AI 误挑了模式（罕见），手动指定"用 /harness-plan"项目初期偶尔，后期几乎不发生打 4 个字
归档老任务时 review 一下 _archived/ 看是否要永久删（非必须）项目末期自由心证

7.3 你减少的事

  • ✅ 不用再为"调研类问题"忍受 30 分钟的完整 7-stage 仪式
  • ✅ 不用为"keep refactoring until..."类目标手动 loop 跑 N 次
  • ✅ 不用每次新任务重新告诉 AI 那些项目特有的坑（insight-index 替你记着）
  • ✅ 不用担心 QA 漏掉的边界 case ——对抗性契约强制 QA 自己找反例

8. 我会在什么时候直接体验到这些变化

场景 1：你说"能不能用 SQLite 替代 Postgres？"
v0.10： AI 默认跑完整 7-stage——写 RA、做架构对比、Gate Review、写代码切换 driver、跑测试、提交。30 分钟后你才发现"其实我只想要一个评估"。
v0.11： AI 识别这是个调研问题，自动用 /harness-explore。5 分钟后你拿到 findings.md：现状分析 + SQLite 的限制 + 不推荐迁移的 3 条具体原因（带 evidence）+ 推荐 next step。
场景 2：你说"看看这个设计有没有问题，先别动手"
v0.10： AI 不理解"先别动手"，跑完整流程；你打断后 7-stage 已经走了一半，状态半残。
v0.11： AI 识别"先别动手"，用 /harness-plan，跑完 RA + SA + GR 三阶段，给你 03_GATE_REVIEW.md 的判决（APPROVED / CHANGES REQUIRED / REJECTED）。你看判决决定下一步——批准就跑 /harness 续上 Dev 阶段（已经存在的 01-03 不重写）。
场景 3：你说"持续重构到 verify_all 全绿"
v0.10： AI 跑一次 7-stage 后停止；你看 verify_all 还有 5 个 WARN，得手动说"再来一次"。每轮都浪费完整流程开销。
v0.11： AI 用 /harness-goal，预算 10 轮 / 60 分钟。每轮 Dev 改一点 → verify_all 跑 → 直到 5 → 4 → 3 → 0 → break out，写 GOAL ACHIEVED。每轮 Dev 只看最近 3 轮历史，不让 context 爆炸。
场景 4：QA 阶段 catch 到一个 v0.10 会漏掉的 bug
v0.10： Dev 写 parseDate(s)，写测试 expect(parseDate('2026-01-01')).toEqual(...)。绿了。QA 顺着读，绿了，放行。生产里 parseDate('2026-1-1')（无 leading zero）崩溃。
v0.11： QA 按对抗契约："AC 是'接受 ISO 日期'，我预期 2026-1-1 这种非严格 ISO 会让它崩。" 自己写 tests/parse-loose.test.ts，跑，挂了，报 BLOCKER。Dev 修复后再跑，绿。生产不挂。

9. 跟原项目（Harness Engineering）哲学的关系

原项目（4 篇文章）核心主张v0.11 怎么对应
"少代码，多文档"三个新模式纯 SKILL.md markdown，没新运行时；对抗验证靠 agent 契约 + verify_all 强制；insight-index 是 markdown 文件
"真相源在仓库里、版本控制"insight-index.md 是 git tracked，跨会话/跨工具一致
"事后验证脚本不可妥协"verify_all 加了对抗段 enforce check
"不要让 AI 做架构决策，要让 Harness 做"四个模式是 Harness 层的路由决策，不是 agent 自己拍脑袋
"AI 注意力衰减"反模式insight-index ≤30 行严格约束；goal mode 每轮只看 3 轮历史
反模式："hooks 滥用"archive-task 是显式命令脚本，不是后台 hook；contracts 靠 verify_all 静态检查而非动态 hook
