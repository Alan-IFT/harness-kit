## Design drift (if any)

One minor, non-behavioral helper-internal deviation (does NOT change the design contract,
flagged for the reviewer's awareness):

- **`DESIGN DRIFT` (minor, internal): byte-exact section extraction uses a temp-file / line-array
  injection instead of bash command-substitution.** §5.2 describes extracting the canonical
  block and re-injecting it. A literal `$(...)` capture in bash strips trailing newlines and
  would lose the section's trailing blank line (the R7 off-by-one hazard). The helper therefore
  extracts the template section to a temp file (bash) / a line array (PS) and injects it
  verbatim via `awk getline` / array concatenation, preserving the trailing blank line so the
  `[heading, next "## ")` slice is byte-exact. This is an implementation detail in service of the
  design's byte-stable-round-trip requirement (§5.4), not a contract change. Verified by the
  round-trip test #9 passing byte-identically in both shells.

No behavioral drift: every AC, exit code, stdout record, gate, and fan-out site matches the
design and the Gate's adjudications (F-1 helper path, F-2 marketplace line 17, F-3 F.1 gets
`language-policy` not `test-language`, F-4 live exempt array authoritative).

## Insight to surface (optional)

PowerShell's `-split "`n", -1` (the `-split` operator with an explicit max-count of `-1`) does
NOT mean "all substrings" — it collapses the result to ONE element, silently defeating
line-splitting; use `.Split("`n")` (the .NET String method) for cross-shell line parity.
· evidence: `.harness/scripts/language-policy.ps1` `Read-Lines` + probe (`$raw -split "`n",-1`
returned 1 vs `.Split("`n")` returned 86 on an 85-LF file).

  inert. Then **confirm the detected language with `AskUserQuestion`**, pre-filled:
  "Detected `<lang>`. Refresh to the current canonical `<lang>` policy? [yes / switch to
  the other language / cancel]". The confirmed value becomes the real `--lang` from step 5.

### Helper contract verification (the wording is accurate, not aspirational)

The new text claims `DETECT|` is emitted unconditionally after `--lang` validation and
before any mutation, and that `--dry-run` suppresses all writes. Confirmed by reading the
helper:

- `language-policy.sh`:

  - `detect_lang` `emit "DETECT|$detected|$source"` at `:169`, called at `:171` —
    unconditional, runs for both dry-run and real.
  - all project-file writes go through `write_or_noop` and are gated by `[[ "$DRY_RUN" == false ]]`
    at `:196` (and `cp "$cand" "$path"` at `:203`); with `--dry-run` the verb prefix is `PLAN`
    (`:113`) and no project file is touched.
- `language-policy.ps1`:
  - `-Lang` validated via `[ValidateSet("en","zh")]` (`:47-48`) at bind + explicit required
    check `:61-64`, BEFORE detect.
  - `Emit ("DETECT|{0}|{1}" ...)` at `:182` — unconditional.
  - all writes (`Write-OrNoop`) gated by `if (-not $DryRun)` at `:210`; `WriteAllText` at `:217`
    runs only in the non-dry-run branch. With `-DryRun` the verb prefix is `PLAN` (`:104`).
