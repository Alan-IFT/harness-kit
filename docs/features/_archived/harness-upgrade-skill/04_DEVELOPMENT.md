## Design drift

None material. Two minor implementation choices worth the reviewer's eye (neither changes the design contract):
1. **`DESIGN DRIFT` (minor):** the helper references the 3 substitution placeholders via piece-wise string assembly rather than literal `{{...}}` tokens, solely so the copied helper does not trip `test-init`'s "no unresolved placeholders" cleanliness scan. The design (§6.2) only specified *which* placeholders to substitute, not the source form; substitution behavior is unchanged. Flagged so the reviewer understands why `Substitute-Placeholders` reads `$o + "PROJECT_NAME" + $c`.
2. The helper inlines `migrate-scripts-layout`'s relocation logic (design §3.2 S1 explicitly chose "inline, do not shell out") and ADDS `migrate-scripts-layout.{ps1,sh}` to its own relocation known-set + refresh-set, so a project always ends with the current migrate helper present (supports B-2(b)/B-3 parity). This matches the design's intent.

## DEFECT-1 fix (stage 6b)

QA's `06_TEST_REPORT.md` filed **DEFECT-1 [MINOR]**: `upgrade-project.ps1` wrote the
`.git/hooks/pre-commit` body **without** a trailing newline, while the bash helper
`upgrade-project.sh` and the canonical `install-hooks.{sh,ps1}` parity-target write it
**with** a single trailing `\n`. Consequence: running the ps1 helper then the sh helper
(or vice-versa) on the same project produced a 1-byte-different hook → one spurious
`HOOK-INSTALL (refreshed)` + a `.bak`. This violates NFR-1 (cross-shell parity).

### The exact change (file:line, before → after)

File: `skills/harness-init/templates/common/.harness/scripts/upgrade-project.ps1`
(then `sync-self`-propagated byte-identically to the dogfood mirror
`.harness/scripts/upgrade-project.ps1`).

Two write sites — the BC-6 fresh-install path and the stock-rewrite path (QA's `:259`/`:274`):

- **Line 262** (fresh-install, "was absent"):
  - before: `[System.IO.File]::WriteAllText($hookPath, $currentHookBody)`
  - after:  `[System.IO.File]::WriteAllText($hookPath, $currentHookBody + "`n")`
- **Line 278** (stock pre-T-007 refresh):
  - before: `[System.IO.File]::WriteAllText($hookPath, $currentHookBody)`
  - after:  `[System.IO.File]::WriteAllText($hookPath, $currentHookBody + "`n")`

`` `n `` is the PowerShell LF escape — a single `\n`, NOT CRLF. The `$currentHookBody`
here-string is unchanged (hook content/logic identical); only the trailing byte differs.
This makes the ps1 output byte-identical to `upgrade-project.sh`'s
`current_hook_file_content="$current_hook_body"$'\n'` (sh `:216`) and to the canonical
`install-hooks.sh` heredoc output. The NOOP-detection branch (`:268`/`:270`) uses
`.Trim()`, which is insensitive to the trailing newline, so a second same-shell run still
correctly NOOPs.

**Bash side left untouched** (QA's evidence confirmed it is the correct parity target;
my own probe of the canonical `install-hooks.sh` on-disk output confirmed it too — see below).
