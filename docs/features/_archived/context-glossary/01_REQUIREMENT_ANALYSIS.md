## 2. In-scope behaviors

Each item is a concrete markdown / template / agent / index change in THIS repo (harness-kit is a tooling+plugin library; "behaviors" are file edits, not app features).

1. A dogfood `CONTEXT.md` exists at the repo root (`c:\Programs\HarnessEngineering\CONTEXT.md`), seeded with this repo's genuine single-context domain terms.
2. The dogfood `CONTEXT.md` follows the reference format exactly: a `# {Context Name}` title, a one-or-two-sentence context description, then term entries each consisting of a **bold term**, a one-or-two-sentence definition, and an `_Avoid_:` line listing synonyms not to use. (Format source: `c:\Programs\_research\mattpocock-skills\skills\engineering\domain-modeling\CONTEXT-FORMAT.md`.)
3. The dogfood `CONTEXT.md` contains only project-specific domain terms (not general programming concepts). The seeded term set is drawn from this repo's real vocabulary; the canonical candidate set is: `frontier`, `pool` (living pool), `ambient`, `partition agent` (`dev-*`), `stage doc`, `verdict`, `insight` (insight-index entry), `rollback`, `dogfood`, `template overlay`, `soft dependency` vs `hard dependency`. The final included subset and any `_Avoid_` synonyms are decided by the developer at authoring time, constrained by items 4-6 and Boundary conditions; this requirement does NOT pre-pick the exact wording (that is content authoring, not design).
4. The dogfood `CONTEXT.md` contains no `_implementation detail_` content (no "how it works", no file paths as definitions) — it is a glossary, per the reference SKILL.md discipline.
5. A generic template seed for `CONTEXT.md` exists under `skills/harness-init/templates/common/` so every project generated by `/harness-init` receives a `CONTEXT.md` seed.
6. The template seed is GENERIC (a skeleton / placeholder-free starter that instructs the generated project to add its own domain terms), NOT a byte-copy of this repo's dogfood `CONTEXT.md`. (Per insight: a dogfooded template asset like `decision-rubric.md` carries generic template content while the repo's own copy carries real content; the two are intentionally not byte-synced. `sync-self` mirrors only the 7 script pairs and does not touch `CONTEXT.md`.)
7. The template seed contains zero unresolved `{{PLACEHOLDER}}` tokens (so `test-init` / `test-real-project`'s recursive `\{\{[A-Z_]+\}\}` scan does not flag it); if a placeholder-name string is ever needed it is assembled at runtime, not written literally.
8. The `requirement-analyst` agent contract (`agents/requirement-analyst.md`) is amended to reference `CONTEXT.md` as a SOFT dependency: the agent reads the project glossary (if present) to use canonical domain terms in the requirement doc and lazily maintains it (records a newly-coined / sharpened term inline), and the reference is written in vague prose that degrades gracefully when `CONTEXT.md` is absent (no hard "run X first" pointer).
9. The `solution-architect` agent contract (`agents/solution-architect.md`) is amended with the same SOFT-dependency reference: read `CONTEXT.md` (if present) for canonical naming of new modules / files / symbols, lazily maintain it, degrade gracefully when absent.
10. The SOFT-dependency wiring in items 8-9 contains no language that makes `CONTEXT.md` a precondition for the agent to function (no setup pointer, no `BLOCKED` on its absence) — it is reference + lazy-maintain only.
11. AI-GUIDE.md gains exactly one index line in its **Memory layer** section pointing at `CONTEXT.md` with a one-line description and a "when to read" trigger consistent with the existing memory-layer entries.
12. `docs/dev-map.md` gains a location entry for `CONTEXT.md` (both the repo-root dogfood file and the template seed path), consistent with the existing dev-map location-table conventions.
13. `.harness/scripts/verify_all` continues to PASS all checks after every change in this task, in BOTH shells (PowerShell + Bash).

## 4. Boundary conditions

1. **CONTEXT.md absent (generated project never seeded its own / user deleted it):** the RA/SA SOFT-dependency reference degrades gracefully — the agents still produce their stage docs; output is merely less terminologically sharp. No agent returns `BLOCKED` for a missing `CONTEXT.md`. (Verifiable by reading the amended agent prose: it must contain a graceful-degradation clause and no setup pointer.)
2. **CONTEXT.md present but empty / header-only (no terms yet):** treated identically to a lazily-created skeleton — the agent reads it, finds no terms, proceeds normally and may add the first term inline. No error.
3. **A term the agent coins conflicts with an existing glossary term:** the RA/SA maintenance prose covers the "challenge against the glossary" habit (sharpen / reconcile), but this round does NOT require an automated conflict check — reconciliation is a documented habit, not a gate.
4. **Template seed scanned by `test-init` / `test-real-project`:** the seed must contain no unresolved `{{PLACEHOLDER}}` (boundary: the recursive placeholder scan over every generated `.ps1/.sh/.md/.json`). A `CONTEXT.md` seed is a `.md`, so it IS in scope of that scan.
4a. **Asset-count baselines:** adding an always-present template asset (`CONTEXT.md` seed) shifts `test-init` / `test-real-project` generated-asset counts; the baseline (`.harness/scripts/baseline.json` and any in-driver count) must be reconciled from a captured run, not hand-edited. (This is an implementation obligation surfaced as a boundary so the developer/QA expect it; it is not a behavior to "design".)
5. **I.6 retired-claim self-trip:** neither the dogfood `CONTEXT.md`, the template seed, the AI-GUIDE.md index line, nor the RA/SA prose may write any I.6 banned-anchor phrase (verified against the live banned list in `verify_all.{ps1,sh}` lines 521-536 — none of the 14 entries concern glossaries / domain terms / CONTEXT.md, so this is a low-risk "do not introduce" constraint, not an active collision). This requirement document lives under `docs/features/` which is an I.6 exempt dir, so it is itself safe.
6. **Doc-size caps:** the AI-GUIDE.md index line must keep AI-GUIDE.md ≤200 lines (I.1); any RA/SA prose addition keeps each agent file under its 300-line cap (I.2 / rule 70); the dogfood `CONTEXT.md` and the template seed are new long-lived docs and should stay tight (glossary discipline already enforces brevity; no formal cap is added).
7. **Max size of CONTEXT.md:** no numeric cap is imposed this round; the "tight definitions, domain-only terms" discipline is the size control. (If a cap is later wanted it is a future I.* WARN — out of scope here.)
8. **Cross-shell symmetry:** because verify_all must PASS in both shells and `test-init`/`test-real-project` run in both, any count/baseline reconciliation must be applied symmetrically (PowerShell and Bash). The `CONTEXT.md` seed is a static asset (no generation logic), so it carries no write-time newline / encoding parity hazard of its own; UTF-8 save is required (consistent with all repo docs).

## 8. Open questions for user (deferred-human: each carries a recommended default)

**OQ-1 — Does the no-new-guard constraint stand, or is a minimal presence-check warranted?**
The author of the reference material does NOT gate `CONTEXT.md`; INPUT.md explicitly excludes a new guard. I found no evidence that the absence of a guard creates a concrete hazard here (a missing `CONTEXT.md` degrades gracefully by design, and the template seed makes it always-present in generated projects).

- (b) Add a presence/format WARN check to `verify_all`.
- **Recommended: (a).** No concrete hazard justifies a guard; adding one contradicts the standing rubric and bumps the check count for no root-cause reason. **Safe to proceed on (a).**

**OQ-2 — Where exactly does the dogfood `CONTEXT.md` live: repo root, or under `docs/`?**
The reference convention is repo root for a single-context repo, and INPUT.md says "at this repo root." But this repo keeps most long-lived docs under `docs/` and memory-layer files under `.harness/`.
- (a) Repo root `c:\Programs\HarnessEngineering\CONTEXT.md` (matches reference convention + INPUT.md wording; most discoverable; the format's single-context rule assumes root).
- (b) `.harness/CONTEXT.md` (groups it with the other memory-layer files: `insight-index.md`, `decision-rubric.md`).
- (c) `docs/CONTEXT.md`.
- **Recommended: (a) repo root.** It is what INPUT.md states, what the reference format prescribes for single-context repos, and the most agent-discoverable location; the template seed must then also land at the generated project's root. **Safe to proceed on (a).**

**OQ-3 — Does shipping a new always-present template asset warrant a `plugin.json` version bump (and the G.4/G.3 stamp consistency that follows)?**
Past tasks that added a template asset or skill bumped the version (e.g. decision-rubric shipped at v0.28.0). This task adds a new generated-project asset, which changes what `/harness-init` produces.
- (a) Bump the patch/minor version (e.g. v0.33.0 → v0.34.0) since generated-project output changes; update CHANGELOG + version stamps; let G.3/G.4 gate it.
- (b) No bump (treat as a documentation-only change under [Unreleased]).
- **Recommended: (a) a minor bump.** A change to what every generated project receives is user-visible and is the kind of change prior tasks versioned; the developer/PM finalize the exact number at delivery. This is a release-stamping call, reversible, and in-scope → proceed on (a) unless the operator prefers otherwise. **Safe to proceed on (a).**

**OQ-4 — How prescriptive is the RA/SA "lazy-maintain" prose: a one-line read-and-use reference, or the fuller maintenance discipline (challenge / sharpen / cross-reference / update-inline)?**
The reference SKILL.md describes a rich active discipline; INPUT.md says the discipline "folds into RA/SA prose this round" but also keeps it SOFT and token-light.
- (a) Minimal: one or two sentences per agent — "read the project glossary if present for canonical terms; record a newly-coined or sharpened term inline; degrade gracefully if absent."
- (b) Fuller: a short bullet list per agent covering challenge-against-glossary + sharpen-fuzzy + update-inline + lazy-create.
- **Recommended: (a) minimal, leaning lightweight.** Token economy (NFR-2) and the SOFT-dependency principle favor the smallest wording that conveys read + lazy-maintain + graceful-degrade; the agent files are near their purpose-density and adding a full discipline block risks the 300-line cap and over-prescription. The fuller discipline can be a future dedicated skill (already out-of-scope item 3). **Safe to proceed on (a).**

**OQ-5 — Multi-context future: mention `CONTEXT-MAP.md` as a future option, or omit entirely?**

- (a) Add one sentence in the dogfood `CONTEXT.md` and/or dev-map noting multi-context via `CONTEXT-MAP.md` is a future option, single-context is current.
