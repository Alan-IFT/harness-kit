harness-kit：用设计消除"老项目拿不到新内容"

设计评审面板(12 agent,核对 Claude Code 插件官方文档 + diff/git ls-files 验证)· 2026-06-10

结论  能设计消除 —— 而且就是你赞过的 T-16 "消除重复"模式,只是规模更大。 根因是:框架内容(agents / 通用规则 / 脚本 / AI-GUIDE)在 /harness-init 时被拷进每个项目。 这份拷贝既是臃肿、也是漂移源。--include-content 是加一套同步去追这份重复 —— 治标。 把"拷贝"去掉,漂移类就不存在了(不是桥接,是删除)。

铁证:你的 15 个 skill 现在已经是这样分发的 —— 零拷贝、零刷新、零漂移。"更新插件 == 每个项目都最新"对 skill 早已字面成立。这个改造只是把同一套已验证的机制延伸到 agents(干净)和规则(过 spike 后)。

1 · 重复有多大(已验证)

框架不变内容 = 漂移/臃肿源

每个项目都被拷一份、且各项目逐字节相同:
  • 8 个 agent ≈ 1074 行 —— diff 验证三处(template/.harness/.claude)逐字节一致,纯三重复制
  • ~8 条发版规则 ≈ 414 行(含刚加的 25-decision-policy)
  • AI-GUIDE 路由 ≈ 97 行
  • 9 对脚本 ≈ 3167 行(脚本是被"执行"的,根本没有 tool-agnostic 理由要拷)

真·项目专属 = 合理本地(不是问题)

  • 50-<type>.md 类型 overlay + 分区 dev-* agent
  • decision-rubric 的 Custom(Mode 3) 段、Active mode 那一行
  • insight-index、tasks.md、docs/features/、baseline.json
  • 00-core 的头部 token + 输出语言选择、settings.json

≈ 90% 是不变内容,<10% 才是项目专属。漂移与臃肿全部在那 90%。(面板原稿把不变行数夸到 ~5000;我核对后改正:真正发版到用户项目的不变散文约 1600 行 —— 规则 10/15/20/30/40/80 是本仓库 dev-only、从不外发,git ls-files templates/ 零命中。agents 仍是最大、最干净的一块。)

2 · Before → After

现在:拷贝模型

插件模板 → 拷进每个项目的 .harness/。从此两份各自演化 → 漂移。新内容(决策策略)只随 init 进新项目;老项目要 /harness-upgrade,但它 v1 明确"不刷新规则内容" → 死圈。

之后:引用模型

框架内容只在插件里一份;项目只留那 ~10% 的 delta + 一个指针。${CLAUDE_PLUGIN_ROOT} 在 skill/agent/hook 体内行内展开,随版本自动重解析。更新插件 == 每个项目都最新,没有"内容刷新"这一步 —— 那一步消失了,而不是被打补丁。

3 · 分三腿落地(按"需证明程度"递增,可中途停)

腿做什么风险 / 是否已验证规模判定
L1 · Agents
先做		 给插件加顶层 agents/ + 在 plugin.json 声明。8 个 agent 三重复制 → 收成插件里一份,namespaced harness-kit:<name>。顺带让 harness-sync 的 agent 同步彻底失去存在理由。 平台已支持 官方文档证实:插件顶层 agents/ 自动发现、按 harness-kit:<name> 命名。单笔最大去重、机制最少。 ~1–2 天
可独立发版		 做(消除)
L2 · 规则 + AI-GUIDE
spike 后		 ~8 条规则 + AI-GUIDE 作为插件资源,经常驻 skill 或(更稳)agent 体内 @${CLAUDE_PLUGIN_ROOT}/… 读取暴露。 需 spike 行内 @-读取是官方机制、但本仓库零先例。静默失败(规则根本没加载)比"旧但在"更糟 → 先在你的 Claude Code 上验一天。 spike ~半天
过则 ~1–2 天		 条件做
L3 · 脚本 / Hook
最后		 脚本引擎 → 插件;guard-rm/ambient/sync 三个 hook → 插件自己的 hooks.json;verify_all 的 pre-commit 入口留一个瘦本地 shim。 有硬约束 关键纠错:${CLAUDE_PLUGIN_ROOT} 不会在项目 settings.json 和 .git/hooks 里展开 —— hook 必须搬进插件 hooks.json,git 钩子只能瘦 shim。 ~1 天 混合

所以这是有序混合而非纯改造:agents 干净消除;规则视 spike;有一小撮本地不可避免(git pre-commit 钩子永远拿不到插件变量 → 瘦 shim 长期保留)。诚实地说,纯加法补丁是错的主答案(它继续拷贝、再加同步去追,正是你反对的"加守卫致臃肿"),但一丝加法逻辑在平台够不到的腿上不可避免 —— 假装没有(像面板初稿把项目 settings 的 hook 路由进一个不展开的变量)就会发出坏东西。

4 · 你要清醒接受的取舍

  • 默认路径放弃 tool-agnostic(对不变层)。 ${CLAUDE_PLUGIN_ROOT}/插件 agent/skill 加载都是 Claude-Code-only,Copilot/Cursor 解析不了。对你(只用 Claude Code)是零成本 —— 拷贝模型的四个理由(Copilot 可读 / 离线 / 每项目 git review 相同字节 / 就地改不变文件)全是对你而言的死重。但 harness-kit 是公开插件 → 这是一次明确的"Claude-native by default"取向,须诚实写进 README,并保留真实、可测的 /harness-init --portable 逃生口给需要的人。
  • 插件耦合 / 失去离线自包含 —— 但你的 15 个 skill 早就强加了同一依赖,不是新风险类。
  • "更新==最新"是最终一致,非原子 —— 当前会话保留旧缓存版本,需 /reload-plugins 或新会话;旧缓存约 7 天后回收。文档要如实写。
  • 一次性迁移会动到本仓库自己的门禁(E.1/E.2/E.4b + Stop-hook 正是在强制"拷贝/同步"模型)—— 必须同笔原子改写,否则本仓库 verify_all 变红。

5 · 需要你拍板的真叉路(附我的倾向)

① 方向 GO? 承诺"Claude-native by default、框架引用不拷贝、tool-agnostic 降级为 --portable 选项"?
我的倾向:Yes。 它符合你"设计消因"原则,你又只用 Claude Code;--portable 对公开用户保持诚实。
② 先做哪步? 先做 L1(agents 外置) —— 安全、最大去重、可独立发版、~1–2 天;规则/脚本之后再说。
我的倾向:先 L1。 它自身成立、且证明整个模型,不必等 spike。
③ 分区 dev-* agent:做成固定插件类型变体(干净零拷贝,但失去 {{STACK}}/{{PROJECT_NAME}} 注入)还是保留为唯一的每项目本地 agent(保注入、留小本地拷贝)?L1 实现时才需定。
④ 规则投递面:常驻 skill(更干净,但模型可能不调用 = 可靠性退化)还是 agent 体内 @-读(agent 跑时必加载,但只到 agent 上下文)?我的倾向:agent 面,更可靠。 L2 才需定。

推荐第一笔: L1 —— agents 外置。删掉每项目最大的一块重复(~1074 行 × 每个项目),杀死 harness-sync 的 agent 腿,且决策策略死圈顺带消解(策略一旦插件常驻,就没有"缺失文件"可言,只剩本地那一行 Active mode)。这是独立可发的安全胜利。
