Harness Kit · 项目详情与使用场景

一个 Claude Code Plugin（10 个 skills + 项目模板），把"有纪律的 AI 驱动开发"装到任何全栈 / 后端 / 通用项目里。 人工只做"描述需求"和"AI 卡住时介入"，其他 — 7-Agent 流水线、验证闸门、结构化文档 — 自动跑。
v0.17.0 verify_all 30/30 test-init 227/227 integration 82/82 MIT Dogfood

一、项目身份

项目本质Claude Code Plugin（工具包，不是框架）
承载形态10 skills + 项目模板
主语言Markdown + PowerShell + Bash
主工具Claude Code（兼容 Copilot 切换）
输出语言策略项目级（init 时选；中文项目全程中文）
特殊性Dogfood — 自己用自己开发

Harness Kit 不重造平台机制（Sandbox、Hooks、Sub-agents、MCP、Memory 是 Claude Code 已有的）， 它只提供内容：把 7 个角色 agent、规则片段、验证脚本、项目骨架打包好， 让任何项目装上就能跑"7-stage 流水线"开发。

定位关键词：工具无关 SOT 层 + 工具特定 binding 层 + 渐进披露（progressive disclosure）+ 对抗性验证 + 跨任务 insight 累积。

二、核心架构

两层模型：SOT vs Binding

.harness/rules/*.md ← 单一真相源（你只编辑这里） ↑ │ 引用 │ AI-GUIDE.md ← 工具无关入口（~50 行索引） ↑ │ 指向 │ CLAUDE.md ~15 行 stub (Claude Code 读) .github/copilot-instructions.md ~15 行 stub (Copilot 读)
Context 预算节省

v0.9.x 全量 CLAUDE.md = ~3500 token 常驻

v0.10 stub 模型 = ~250 token 常驻

小问答 ~92% 节省 · 中等任务平均 ~50% 节省

规则片段按需懒加载

每个 .harness/rules/*.md 顶上写"什么时候读"

AI 跟着 AI-GUIDE.md 的触发条件，只加载相关 1-3 个片段，不全量装载。

三、4 种执行模式（同一项目里按任务形态选）

Skill 代价 什么时候用 什么时候不用
/harness 100% 基线 真实功能 / bug / 重构，有清晰验收标准 调研、可行性、纯设计、循环改进
/harness-plan ~35% 投入工程时间前先验证设计（RA → SA → GR 停） 已经知道怎么做 → 直接 /harness
/harness-explore ~15% "能不能做" / "库 Y 行不行" — 出 findings.md 已经决定要做 → 走 plan 或 harness
/harness-goal 预算限定 "循环改进直到 X"（覆盖率到 80%、verify_all 全绿） 有清晰单次定义的任务
反模式：单行 typo / 改注释 → 直接编辑 + 跑 verify_all，不要走 7-stage。 误用流水线 = 把简单事情仪式化。

四、两类项目的使用方式

🟢 场景 A · 新项目（空目录 / 刚 git init）

从零开始 — 让 AI 从第一天就以 7-Agent 纪律开发。

具体例子：用 Next.js + NestJS + Postgres 起一个订单管理系统

  1. 建空目录 + 启动 Claude Code 
    mkdir order-system && cd order-system
git init
claude
  2. 调 init skill 
    /harness-kit:harness-init
    AI 弹 AskUserQuestion，问 5-6 个问题（弹窗式，不用敲字）：
    • 项目类型 → Fullstack
    • 技术栈 → Next.js + NestJS + Postgres（自由文本）
    • 装 auto-sync hooks → Yes
    • Developer 分区 → Partitioned（dev-frontend / dev-backend / dev-db）
    • 输出语言 → 中文
    • AI 草拟项目专属规则 → Yes（v0.16+，AI 根据栈描述写 50-order-system.md）
  3. ~30 秒后 项目里多出： 
    order-system/
├── .harness/                  # 工具无关 SOT
│   ├── agents/                # 7 个标准 + 1 supervisor
│   ├── rules/                 # 含 AI 草拟的 50-order-system.md
│   ├── skills/
│   └── insight-index.md       # 跨任务 hard-won truths（≤30 行）
├── .claude/                   # Claude Code binding
├── .github/copilot-instructions.md  # Copilot binding stub
├── CLAUDE.md                  # ~15 行 bootstrap stub
├── AI-GUIDE.md                # 工具无关入口
├── docs/                      # workflow / dev-map / tasks / features/
├── scripts/                   # verify_all / harness-sync / install-hooks
└── evals/                     # 金标准回归任务
  4. 开始描述需求（用自然语言，PM 自动接手） 
    用户："在订单页加一个 CSV 导出按钮，支持按日期范围筛选。"

→ PM Orchestrator 自动路由 7 stage
→ Architect 给出 partition 分配（dev-db → dev-backend → dev-frontend）
→ 6 份阶段文档自动写到 docs/features/csv-export/
→ 跑 verify_all PASS
→ archive-task 把 insight 收割到 .harness/insight-index.md
→ docs/tasks.md 标 stage:done
  5. 中途想介入（v0.13+）→ 写一条软指令，PM 在下一个阶段边界消费 
    /harness-kit:harness-intervene "REDIRECT: 用 streaming export 而不是一次性 dump"
关键收益：从第一行代码开始，结构化文档 + 验证闸门 + 跨任务知识累积全部就位。 后续任何"看起来简单但容易翻车"的事都被规则拦着，比如：QA 必须写对抗性测试、下游不能改上游、verify_all 不绿就不算完。

🟣 场景 B · 已开发中项目（有代码、测试、CI 但没 Harness）

非侵入接入 — 不重写、不替换、不破坏现有约定。

具体例子：一个已经跑了 1 年的 FastAPI + Postgres 项目，团队想引入 AI 纪律

  1. cd 到项目根目录，启动 Claude Code 
    cd /path/to/existing-fastapi-project
claude
  2. 调 adopt skill 
    /harness-kit:harness-adopt
    和 init 的区别：adopt 先侦察，再问，再 apply。
  3. AI 自动侦察（不写任何文件）：
    • 读 pyproject.toml → 检测到 FastAPI、SQLAlchemy、alembic、pytest
    • 读 .github/workflows/ → 检测到 CI 用 GitHub Actions
    • 读 README.md / CONTRIBUTING.md → 提取既有约定
    • 看目录结构（无 apps/web）→ 推断为 backend
    产出一份"侦察报告"打到屏幕。
  4. 用户确认推断值（AskUserQuestion 弹窗）
    • 类型确认：Backend ✅
    • 是否启用分区（dev-api / dev-services / dev-db）→ Yes
    • 输出语言 → 中文
    • 是否让 AI 草拟项目专属 50-<slug>.md 规则（基于已读到的代码约定）→ Yes
  5. 冲突处理（关键差异点）：
    • 如果项目已有 CLAUDE.md / .harness/ → AI 询问：[A] Cancel / [B] Merge / [C] Overwrite
    • Merge 模式 → 只补缺失的，绝不动既有文件
    • Overwrite 模式 → 每个冲突文件单独再问一次
  6. Apply 阶段（用户明确 "yes" 后才动文件）
    • 写入 .harness/（含 AI 草拟的 50-<slug>.md，每段带 <!-- source: ... --> 来源标注）
    • 写入 .claude/ + CLAUDE.md stub + copilot-instructions.md stub
    • 生成 scripts/verify_all — 自动接到项目已有的 pytest / alembic check 上，不替换
    • 不动 src/、tests/、migrations/ 任何一个字节
  7. 第一次试跑 
    用户："优化 /orders 列表接口的 N+1 查询。"

→ PM 路由 7 stage
→ Architect 读项目 dev-map.md（adopt 时已根据真实代码生成）
→ dev-services / dev-db 分区按 Architect 指派工作
→ QA 写对抗性测试（包含 explain analyze 验证、压力测试）
→ verify_all 串上项目原有 pytest，全绿才算完
关键收益：保留所有现有代码和测试；规则文件 50-<slug>.md 基于真实代码起草， AI 后续操作都会遵守项目本身的约定（比如已有的 lint 规则、命名约定、目录分层），不会强加 harness 的偏见。
注意：adopt 不会自动把已有任务塞进 7-stage 流程；老 issue / 老 PR 不动。 流水线只对"接入后新发起的任务"生效。

五、新 vs 已有项目 — 核心差异对照

新项目 (/harness-init) 已开发项目 (/harness-adopt)
前置条件 空目录或刚 git init 已有源码 / 测试 / package manifest
是否侦察现状 否 — 直接问 是 — 先扫栈、CI、测试、既有约定
问题数量 5-6 个（一次性弹窗） 侦察后再 1 次 AskUserQuestion 让用户确认推断值
verify_all 接什么 模板版（npm test / pytest 占位 — 你后续补） 自动接到项目已有的 lint / test / migration check
50-<slug>.md 规则来源 AI 根据 Q2 stack 描述 + 顶层文件名 起草 AI 额外读项目已有 README/CONTRIBUTING 提取约定
冲突处理 遇到已有 .harness/ 就停（叫你跑 adopt） 三选一：Cancel / Merge / Overwrite，每个冲突文件再确认
对既有代码影响 无（本来就是空） 零字节修改 — 仅添加新文件
第一次任务可用时间 init 完成后立即 adopt apply 后立即；既有任务不溯及既往

六、装这个 Plugin 的三种方式

方式 1 · Plugin Marketplace 推荐
/plugin marketplace add Alan-IFT/harness-kit
/plugin install harness-kit@harness-kit-marketplace

命令带 namespace：/harness-kit:harness-init

方式 2 · 一行 curl / PowerShell
# macOS / Linux
curl -fsSL https://raw.githubusercontent.com/Alan-IFT/harness-kit/main/install.sh | sh

# Windows
iwr -useb https://raw.githubusercontent.com/Alan-IFT/harness-kit/main/install.ps1 | iex

复制到 ~/.claude/skills/，命令无 namespace

方式 3 · 本地 clone（开发模式）
git clone https://github.com/Alan-IFT/harness-kit ~/harness-kit
~/harness-kit/install.sh
~/harness-kit/install.sh --project .  # 项目级
~/harness-kit/install.sh --dry-run    # 预览

七、设计原则（八条红线）

1. 不重造平台机制 — Sandbox/Hooks/Sub-agents/MCP/Memory 是 Claude Code 的事
2. 机制 vs 内容 — 平台给机制，本仓库给内容
3. 工具无关 SOT vs binding 层 — .harness/ 真相；其余生成
4. 演化式交付 — MVP → Hardening → Scale，不大爆炸
5. 基线只升不降 — 测试数、规则覆盖永不悄悄退步
6. 发现者不修 — Reviewer 不改代码，Gate 不改需求
7. 下游不能改上游 — 只能通过 PM 提阻塞回退
8. PM 只路由 — 永不给专业判断
