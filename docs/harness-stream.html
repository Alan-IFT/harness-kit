harness-stream

流式 / 活池开发模式 —— “我只提需求、看结果，AI 自己规划顺序并尽力跑完所有任务”

v0.33.0 skill: /harness-kit:harness-stream 复用 batch 的 BATCH_PLAN.md 池格式 verify_all 32/32（未加新检查）
一句话：它是 /harness-batch 的兄弟模式 —— 同样把任务一条条灌给 7 阶段流水线, 但任务池是 活的:每轮迭代都重读 BATCH_PLAN.md,所以你运行中追加的任务会被自动规划执行, 无需重新调用;而且采用尽力完成语义 —— 某个任务失败只标记+跳过,整条流不停。 自 v0.32 起,复杂需求入池时自动拆解成多条带依赖的子任务(简单需求仍是一条;你手写的行原样执行); 自 v0.33 起,需要人协助的任务被延后而非停流 —— 设为 needs-human 状态、记下诉求、只挡它自己的下游,其余照常跑完,所有待办在结束时一次性汇总。

1 · 你的诉求 → 为什么需要一个新模式

诉求场景:开发过程中不断发现新问题 / 冒出新需求,不想等当前会话跑完再提,而是马上在聊天框发,让 AI 自己规划,我只提需求、关注结果。

现有 /harness-batch 在两点上和这个诉求刻意相反,所以不是“缺功能”,是设计取向不同:

batch 的设计

冻结计划

启动时一次性解析 BATCH_PLAN.md 建好拓扑序,循环里不回读。中途加的行本次看不见 —— 反模式明确写着“停批 / 改计划 / 重新调用”。

batch 的设计

失败即停

第一个硬失败就停整批,且永不自动重试。适合“已知清单 + 快速失败”,不适合“尽量把所有任务都跑完”。

2 · 必须先懂的硬约束(它决定了方案形状)

Claude Code 的 turn 队列: 当 AI 正在一个 turn 里忙(连续派发子 agent)时,你在聊天框发的消息会排队,直到当前 turn 结束才被读到。 所以“马上发、马上被消费”能不能成立,取决于一个 turn 有多大:
  • 一个 turn = 整批 N 个任务(batch 的 fire-and-forget)→ 你得等整批跑完 ✗
  • 一个 turn = 一个任务(stream 在 /loop 下)→ 你最多等当前这一个任务,分钟级 ✓

因此 stream 提供两条输入通道,你按场景选:

文件通道(瞬时,永远可用)
往 BATCH_PLAN.md 追加一行,或发一条 ADD <slug> — <goal> 干预。 循环每轮迭代顶部都重读这两处,所以文件一写就在下一个任务边界被捡起 —— 哪怕在单个连续 run 里也成立,不依赖任何外部能力。
聊天通道(下一轮,符合你的直觉)
直接在聊天框发需求。在 /loop 驱动下每轮是独立短 turn,所以你发的消息在下一轮被吸收并镜像进任务池。 这是最接近“随时发、AI 自动折叠进来”的体验。

3 · 运行模型

    你(聊天框 / 文件)──提需求──▶  任务池  docs/batches/<id>/BATCH_PLAN.md
                                        │   (ID | Slug | Goal | Mode | Depends on | Status)
        ┌────────────  每轮一个迭代(在 /loop 下 = 一个 turn) ───────────┐
        ▼                                                               │
  ① 吸收新需求   重读 BATCH_PLAN.md + 查 intervention(ADD/STOP/NOTE/SKIP);仅 /loop 下额外吸收本轮聊天消息;复杂需求按“入池分诊”自动拆成 N 条;歧义不阻塞,记为 needs-human 待澄清
  ② 重新规划     校验表 → 按 Depends on 重算拓扑前沿(已 DELIVERED 的跳过 = 免费续跑)
  ③ 取下一个     topological frontier 里挑一个 ready 任务
  ④ 派发         Task → pm-orchestrator(mode=full,独立子上下文)   ← 绝不绕过 PM
  ⑤ 回归门       verify_all:FAIL = 硬急停;PASS = 继续
  ⑥ 尽力完成     DELIVERED→done;FAILED→标记 failed + 依赖链标 blocked;需要人协助→标 needs-human + 记诉求 + 只挡其下游;然后继续
  ⑦ 汇报         一行状态:<id> <verdict> · 队列 N pending / F failed / B blocked / H needs-human
        └──────────────────────── 下一轮 ───────────────────────────────┘
  退出:池子清空 且 连续 K 轮(默认3)无新需求 → 写 STREAM_REPORT.md(开头先列 ## Needs your input 待办)

三类结局 —— 失败跳过 · 需人延后 · 硬急停

best-effort(任务级失败)

某任务 FAILED → 标记 + 跳过其依赖 → 整条流继续跑别的任务。这是和 batch 最大的不同,服务“尽量把所有任务跑完”。

needs-human(延后,不停流)· v0.33

需要人协助 → 设 needs-human 状态 + 记下确切诉求 + 只挡它自己的下游 → 整条流继续。涵盖:需澄清、当前决策模式保留给人的判断、请求执行安全敏感动作(只记录,绝不擅自执行)。所有待办在结束时汇总到 STREAM_REPORT.md 开头的 ## Needs your input,你忙完一次性处理。

hard stop(仍然停)

这些会停整条流,因为继续会污染后续:
• verify_all FAIL(基线被打坏)
• intervention.md 里的 STOP
• 安全 hook 拦截了破坏性命令

分界:请求执行安全敏感动作 = needs-human 延后(不执行、不停流);安全 hook 实际拦下一条破坏性命令 = 硬急停。

4 · 三种顺序模式怎么选

维度/harness/harness-batch/harness-stream
任务数量1 个固定的已知清单开放、运行中持续增长
运行中加任务不适用✗(需停批重调)✓ 每轮重读池
失败处理回滚/上报失败即停整批标记+跳过,继续
需要人协助当场询问当场询问 / 停延后(needs-human),继续;结束汇总
硬安全急停✓✓✓(同 batch)
每任务引擎都是 pm-orchestrator(7 阶段,独立子上下文)—— stream/batch 只是上面的薄循环
典型触发“加个功能 X”“把 T-01~T-09 批量跑了”“边开发边丢需求,只看结果”

5 · 怎么用

推荐:聊天驱动(hands-off)
/loop /harness-stream my-pool

每轮处理一个任务,turn 之间吸收你聊天框里新发的需求。你只管提需求、看每轮的一行汇报。

默认:连续 run(零依赖)
/harness-stream my-pool

单次调用内部循环,每轮重读任务池。靠文件通道加任务(追加行 / 发 ADD 干预),瞬时生效。/loop 不可用时用这个。

运行中插入一个任务(任一即可)

  • 聊天: 直接说“顺便加个任务:给订单页加 CSV 导出” → 下一轮被吸收并写进池
  • 文件: 往 docs/batches/my-pool/BATCH_PLAN.md 追加一行 pending
  • 干预: /harness-intervene 写 ADD orders-csv — 给订单页加 CSV 导出
使用前提: /harness-stream 自 harness-kit v0.22.0 起提供;入池分诊(v0.32)与需人延后(v0.33)是后续增强。 若你装的插件版本偏旧,在 Claude Code 里跑 /plugin 把 harness-kit 更新到 ≥0.33.0,即可用上本页描述的全部行为。
