决策记录 · 暂缓实现(A、B 均不建) v2 · 已过 3 路对抗评审(git/env · 并发 · 现实性) 底座:v0.22.0 串行 stream(已交付 2e134ea)
最终决策(2026-06-06,依据「好 UX · 软件工程标准 · 长期易维护」原则):暂不实现并行(A、B 均不建)。
  • 日常用串行 v0.22.0 + 已免费的「任务内 partition 并行」。 这已满足「丢需求→AI 调度→跑完」的原始需求,且零 footgun。最好的 UX 是简单可靠的路径,最易维护的是最小的代码面。
  • 并行对很多项目是「净亏」 —— 集成/有状态测试必须串行(Amdahl),叠加 token×N 和合并冲突重试,常常省不到时间却多花几倍 token。先算账(§2)。
  • Model B(同树 partition)= 按需路径。 比 A 安全(无分支无合并,因此没有合并活锁那个循环),但仍需 Scope 纪律且受 Amdahl 封顶。仅当攒到一批真解耦、§2 账算得过来的任务才建。
  • Model A(worktree 真并行)= 搁置(风险 > 收益)。 env 供给 / Windows junction 非管理员失败 / 每任务分支提交破坏现流程 / 合并活锁需整套调度+集成协调层。你的机器是 Windows 11 Home,本就过不了 junction 那关。

1目标 / 非目标

目标
  • 从 ready 前沿自动识别"可同时做"的任务,并发开发,缩短墙钟。
  • 不牺牲质量门(集成回主干仍过完整 verify_all)。
  • 不牺牲安全(任务间不互相覆盖代码)。
  • 失败仍 best-effort。
非目标(本期不做)
  • 并行有耦合的任务;并行跑有状态集成/e2e 测试(一律串行)。
  • 替项目解决测试基础设施(隔离 DB / 容器编排)。
  • 自动探测 hermetic 测试(只接受显式声明)。
  • 跨机器分布式。单机。

2先算账:并行什么时候是净亏

这是评审标记的 blocker —— 必须诚实量化,否则会误导你开了并行反而更慢更贵。

墙钟模型(N 个独立任务):
  串行总时长  =  N × (开发 + 全量验证)
  并行总时长  ≈  max(各任务开发 + hermetic 验证)  +  N × 串行集成段(merge + 全量 verify_all + 有状态测试)
                 └── 能并行的部分 ──┘            └──────── 永远串行(Amdahl 下限)────────┘

worked example(3 个任务,典型带 DB 的后端):
  每任务:开发 40min,hermetic 单测 2min,全量验证(含集成/DB)10min
  串行:        3 × (40+10)        = 150 min
  并行(N=3,0 冲突): 42 + 3×10    ≈  72 min     → 省 ~52%,但 token ≈ 3×
  并行(1 次冲突重试 +15min):      ≈  87 min     → 省下降
  若 hermetic 只占验证 10%、集成 90%:串行段主导 → 几乎不省,白烧 3× token
判据(强烈建议先自测):
  • hermetic 测试占总验证时间 < ~20% → 串行段主导,并行不值 → 用 Model B 受限并行 或 干脆串行。
  • 任务间 interdependent 文件多 / Scope 难划清 → 冲突重试抹掉收益 → 串行。
  • 按 token 计费 / 预算紧 → 并行 N× token 成本,除非确证净正收益,否则串行。
  • 真正解耦(前端页 + 独立后端接口 + 独立脚本)+ 单测多 + token 不敏感 → 并行才划算。

4两条路线(都不是免费午餐)

Model B · 推荐做 v0.23.0 MVP

partition 同树并行

同一个工作树,按架构师声明的非重叠 partition 并发编辑不同文件(复用现有 dev-* partition 机制),然后一次 verify_all 跑合并后的树。

✓ 无 env 供给问题(共用一套 .env/DB)·无 Windows junction 问题 ·无每任务分支/提交改造 ·复用现有代码

⚠ 无 git 隔离 → Scope 划错会静默互相覆盖(没有冲突这道安全网)→ 必须强制 Scope(§8)·有状态测试仍串行

Model A · 推后到 v0.24+(go/no-go)

worktree 真并行

每任务独立 git worktree + 分支,同时开发,串行合并集成。

✓ 任意任务级并行 ·git 隔离会把隐藏耦合暴露成 merge 冲突(有安全网)

✗ env 供给(§6)·Windows junction 需管理员 ·每任务分支提交是对 harness 流程的破坏性改动 ·pm-orchestrator worktree 子模式尚不存在(§9)

关键反直觉: A「env 难但耦合安全」(冲突能被 git 抓到);B「env 易但耦合不安全」(靠 Scope 纪律,划错就静默损坏)。两者都绕不开 C3:有状态测试一律串行。

5Model B 设计(MVP)

while 池里还有可推进的 pending:
    ingest()                                   # 复用串行 stream:重读池 + intervention(ADD/STOP/…)
    ready = 拓扑前沿
    group = 贪心挑「Scope 两两不重叠」的子集(上限 N)    # Scope 必填且校验,见 §8
    if group 为空但 ready 非空: group = [ready[0]]        # 全重叠 → 退化串行
    并发派发 group 各任务的 Dev(同一工作树,各改各的 partition 文件)   # 复用 pm partition 并行
    barrier:等 group 全部完成编辑
    【串行集成】对该 group 跑一次 全量 verify_all(含有状态/e2e)
        PASS → 该 group 各任务 done;(编排器串行)archive + 写 insight/tasks.md
        FAIL → 二分/逐个定位坏任务:标 failed + 依赖者 blocked,其余保留;best-effort 继续
退出 → STREAM_REPORT.md(§10)
  • 并发的只是「编辑不同文件」;构建/测试/verify_all 在 barrier 后串行跑一次(C3)。
  • 失败定位:一个 group 的合并验证 FAIL 时,因为在同一棵树,需要逐个或二分回滚定位坏任务(B 的代价之一)。
  • 元数据写入(archive/insight/tasks)只在串行段由编排器做(C5)。

6环境策略 —— 你的核心问题(主要约束 Model A)

新建 worktree 里 git worktree add 只 checkout tracked 文件;gitignored 的全不在。Model B 同树没有这个问题;以下是 Model A 必须解决、也是把它推后的原因之一。

类别例子Model A 内怎么办失败模式 / 注意
只读可共享 node_modules/venv/纯配置 symlink(*nix)/ junction(Win);或快速重装 npm ci。不拷贝。 OQ-1 venv 绝对路径 / pip install -e . editable 跨 worktree 易碎;npm 解析 realpath 也可能出错。
有状态(被测写) datas//本地 DB/端口/Redis/上传目录 不进 worktree。 需要它们的测试推到串行集成门在主干跑。 这是 worktree 并行的真正边界(C3)。
密钥 .env API_KEY / DB_PASSWORD 只用 tracked 的 .env.example 在 worktree 合成测试档;绝不 symlink 真 .env(密钥泄漏)。 OQ-2 需要真密钥的测试一律标 integration/e2e,推到串行门。新增 verify_all 检查:.env 的 key ⊆ .env.example。
Windows 现实(你的机器): Win11 Home 下 junction/symlink 非管理员常失败。Model A 必须有预检脚本verify-worktree-readiness:测 symlink/junction 能否建,失败则禁用 --parallel(A)并提示改用 Model B。这是把 A 推后的直接原因之一。

7测试拆分 + 硬前提

并行下"每任务跑完整 verify_all"在隔离里做不到/不划算 → per-task 门拆两段:

隔离内 · 可并行

hermetic-test

无外部状态:纯单测、类型检查、lint、编译。给快速反馈。OQ-3 由项目显式声明命令(如 pytest -m "not integration"),不自动猜;无声明则只编译+lint。

主干集成门 · 串行

全量 verify_all + 集成/e2e

需要真 .env/DB/服务的一切,合并后一次一个任务跑。权威质量门。

硬前提(评审 blocker): 并行模式要求项目声明一个 hermetic 子集且占验证时间 ≥ ~20%;否则并行省不到、应直接串行。技能入口要做前置检查:无 hermetic 声明 / 占比过低 → 警告并建议串行,把开并行变成有意识的选择,而不是默认陷阱。

8独立性 + Scope 强制(B 的安全网在此)

两任务可并行 ⇔ 无依赖边 且 Scope 不重叠 且 无共享有状态资源声明冲突。评审指出 Scope 必须强制,否则是 cargo-cult:

  • 池新增必填 Scope 列(glob 或 partition:dev-frontend),写进 BATCH_PLAN.md 模板。Solution Architect 拆任务时填。
  • Gate Reviewer 拒绝空 Scope;派发时校验 files_changed ⊆ Scope,越界判 DESIGN DRIFT。
  • 冲突预测(评审 blocker):派发 group 前,对各任务做文件级 overlap 预测(B:Scope glob 交集;A:dry-run 三方合并)。预测冲突 > ~30% → 该 group 降级串行。
  • 冲突即证据:真发生重叠/冲突 → 回写 insight-index(「任务 A、B 在 X 文件耦合」),自动加宽 Scope,下次不再并行这对。运行多批后冲突率 >15% → 自动关并行。

9Model A 设计 + 并发正确性(为何推后)

评审在并发维度挖出 6 个 major,集中说明 A 为何"不是改个调度就能上":

问题修法(进入 A 的正式设计)
pm-orchestrator 无 worktree 子模式(OQ-5,破坏性):现状 Developer 改主树、不按任务提交。A 要"在指定 worktree 内运行 + 提交到任务分支 + 不碰主干 + 只跑 hermetic"。新增 mode_variant = worktree-hermetic:阶段 6 只跑 hermetic;不调 archive-task;提交到 worktree-<slug>-<ts> 分支。需单独设计文档 + 改 pm 契约与阶段门(stage-5 门改成"hermetic PASS + 全量延后到集成门")。
元数据 lost-update 竞争(C5):并发 worktree 调 archive-task 写同一 .git 下的 insight-index/baseline,后写覆盖先写。HARNESS_WORKTREE=1 守卫:worktree 内 pm 禁止 archive-task,只把 ## Insight 写进 07_DELIVERY;集成门合并后串行 harvest。
逻辑回归(非文本冲突):A 先合,B 隔离里 hermetic 过但 A 改动破坏了 B 的假设。OQ-6 集成门全量 FAIL 后,在临时 worktree 跑 rebase+hermetic:过 → 逻辑冲突(B 回退 rebase);不过 → B 真 bug。区分以便定位。
分支生命周期:命名/清理/rebase 重试未定义。命名 worktree-<slug>-<ts>;合并成功删 worktree(保留分支供调试);冲突→新 -r2 分支基于新主干 rebase 重跑;退出时 git worktree remove --force 兜底清理。
STOP 语义:并行批中途收到 STOP。OQ-7 优雅停:不再派新 worktree,让在飞的跑完当前任务,未派发的标 skipped,写报告。
dev 阶段失败(非冲突):worktree 内编译/逻辑直接挂。标 failed + 移除该 worktree,不进合并;best-effort 继续;退出统一清理残留 worktree。

并发上限 N 建议 2–4(Workflow 引擎硬上限 min(16,核数-2),实际取小)。这些都解决了,A 才谈得上 v0.24+ 落地。

10可观测性 + 成本(评审补的两块)

可观测性

N 路交错,"只看结果"会乱。STREAM_LOG.md 每任务一行结构化:[ts][task-id][status][elapsed][verify结果][冲突?],展示按 task-id 排序成稳定表(不按完成顺序)。STREAM_REPORT.md 加"并行效率"段:实际墙钟 vs 串行模型、冲突数、各任务 token 花费、是否净正。

成本

N 个全流水线并发 ≈ N× token。报告里逐任务记 token,让你事后判断这次并行值不值。技能入口在 token 预算紧 / 按量计费时主动建议串行。
